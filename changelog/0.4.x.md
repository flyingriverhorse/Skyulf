# Changelog — 0.4.x Series

## v0.4.0 — Parallel Experiment Execution (Phase 2)

### 🔧 Backend
- **Pipeline Partitioning:** New `partition_parallel_pipeline()` in `graph_utils.py` splits a pipeline into independent sub-pipelines for parallel execution. Supports two scenarios: multiple training nodes on canvas, or a single training node with `execution_mode=parallel`.
- **Hybrid Partitioning:** When multiple training nodes exist AND one has `execution_mode=parallel` with multiple inputs, the partitioner now correctly splits that terminal into per-input sub-pipelines while keeping other terminals as single jobs. Previously this hybrid case only created one job per terminal.
- **Multi-Job API:** `POST /api/pipeline/run` now returns `job_ids: List[str]` alongside backward-compatible `job_id`. Each parallel branch creates its own separate job with independent artifact tracking.
- **Branch Ancestry Tracing:** `_collect_ancestors()` BFS-walks backwards from each terminal to include only necessary nodes in each sub-pipeline, preserving shared prefix nodes.
- **Target Node Filtering:** When `target_node_id` is set, the partitioner output is filtered to only the sub-pipeline containing that node. Individual Train buttons run only their branch; omitting it runs all branches.

### 🎨 Frontend
- **Merge/Parallel Toggle:** Both BasicTrainingSettings and AdvancedTuningSettings show a Multi-Input Mode toggle (Merge vs Parallel) when the node has 2+ incoming connections. Mode is user-controlled — no auto-detection.
- **Context-Aware Badge:** Training/tuning node headers show amber GitFork icon when `execution_mode` is explicitly set to `parallel`; all other multi-input nodes show blue Merge icon.
- **Run All Experiments Button:** Toolbar shows a "Run All Experiments" button (Rocket icon, amber) when 2+ training/tuning nodes are connected to separate branches. Queues all experiment branches in one click.
- **Parallel Job Feedback:** Training and tuning submission handlers show the number of parallel branches launched.
- **Pipeline Serialization:** `pipelineConverter.ts` now passes `execution_mode` for both basic training and advanced tuning nodes.

### 🧪 Tests
- **11 new tests** in `test_parallel_partitioning.py` covering: ancestor collection, no-partition cases, multiple terminals, shared prefixes, parallel mode splitting, long branches, metadata propagation.

### 🎯 UX
- **Node Copy-Paste:** Ctrl+C / Ctrl+V copies selected nodes (and their internal edges) and pastes them with a position offset. Supports multi-select and incremental paste offsets.
- **Branch Edge Coloring:** Edges belonging to a single parallel branch are rendered in a distinct solid color. Colors are generated dynamically using golden-angle HSL spacing — unlimited unique colors with maximum contrast, no fixed palette.
- **Branch Labels:** A floating label like "Branch 1 · Random Forest" appears once per branch, positioned near the terminal (training) node — not on every edge. Styled with the branch color and a subtle glow.
- **Naming Consistency:** Renamed "Advanced Tuning" / "Model Optimization" display labels to "Advanced Training" across all frontend surfaces — canvas node, job history tabs, experiments page filter, and job drawer.
- **Feature Importance Tab:** ExperimentsPage now has a "Feature Importance" tab (next to Visual Comparison, Detailed Metrics & Params, Model Evaluation). Shows a horizontal bar chart comparing the top 15 features across all selected jobs. Tab is conditionally hidden when no selected job has feature importance data.
