# Changelog — 0.6.x Series

## v0.6.0 — Ensemble models (Voting / Stacking)

First release of the **ensemble meta-model** family. Combine several base
learners into one estimator, for both classification and regression, directly
from the existing training node.

### 🆕 skyulf-core — Ensemble nodes

- **New `modeling/ensemble.py`** with four registered nodes (Calculator + Applier
  each), kept in their own module separate from `classification.py` / `regression.py`:

  | Registry id | sklearn class | Task |
  |---|---|---|
  | `voting_classifier` | `VotingClassifier` | classification |
  | `stacking_classifier` | `StackingClassifier` | classification |
  | `voting_regressor` | `VotingRegressor` | regression |
  | `stacking_regressor` | `StackingRegressor` | regression |

- **Advanced tuning & per-model hyperparameters in ensembles:**
  - Ensembles now support **Basic mode with fixed hyperparameters** per base model (e.g., custom `n_estimators` for a random forest) and the final estimator (for stacking).
  - Ensembles now support **Advanced/Tuning mode**, running through hyperparameter optimization with search strategies like Random, Grid, Optuna, Halving Random, or Halving Grid.
  - Added support for **nested parameter tuning** inside Advanced mode, auto-generating search spaces for all chosen base estimators (e.g. `random_forest__n_estimators`) and the stacking final estimator.
- **Selectable base learners.** The frontend sends `base_estimators` as a list of
  string keys (e.g. `["random_forest", "logistic_regression"]`); a shared resolver
  (`_BaseEnsembleCalculator`) turns them into `estimators=[(name, instance), …]` —
  mirroring the `base_estimator` pattern in `CalibratedClassifierCalculator`. Unknown
  keys are skipped with a warning; an empty selection falls back to sensible defaults
  so the node always trains. Stacking additionally resolves a `final_estimator` key.
- **Classification base models:** logistic regression, random forest, extra trees,
  gradient boosting, hist gradient boosting, AdaBoost, decision tree, Gaussian NB,
  SVC (probability-enabled), KNN — plus XGBoost / LightGBM when those optional wheels
  are installed (auto-detected only, omitted from the manual picker).
  **Regression base models:** linear regression, ridge, lasso, elasticnet, random
  forest, extra trees, gradient boosting, hist gradient boosting, AdaBoost, decision
  tree, SVR, KNN — plus XGBoost / LightGBM when available.
- **Cross-validation semantics** (the important bit):
  - **Voting** does *no* internal CV — each base model is fit once, then predictions
    are averaged (soft = mean proba, hard = majority vote) / averaged (regression).
  - **Stacking** uses an internal `cv` (default 5) to generate the base models'
    out-of-fold predictions for the final estimator. This is *required* — without it
    the meta-learner trains on in-sample predictions and over-fits. Same shape as
    `CalibratedClassifierCV`'s internal `cv`.
- **Tests:** `skyulf-core/tests/test_ensemble.py` (8 tests) — registration, soft-proba
  range, hard-vote labels, stacking fit with `cv`, regression predictions, unknown-key
  skip, and empty-selection fallback.
- **Tuning Search Strategy Settings Modal:** Fully integrated `StrategySettingsModal` within the advanced Hyperparameter Tuning option panel, letting users fine-tune halving configuration or optuna search strategy settings inside the ensemble configuration interface itself.

### 🧮 skyulf-core — Hyperparameters

- **New `hyperparameters/_ensemble.py`** with per-node field lists, wired into
  `MODEL_HYPERPARAMETERS`, `DEFAULT_SEARCH_SPACES`, and `GRID_SEARCH_SPACES`.
- **New `multiselect` field type** for `base_estimators` (list of base-model keys);
  `voting` (soft/hard) for the voting classifier; `final_estimator` + `cv` for stacking.
- Tunable search dimensions exposed: `voting` (`soft`/`hard`) and stacking `cv`
  (`[3, 5, 10]` default / `[3, 5]` grid). Base-estimator *inner* params are not yet
  auto-expanded into nested `<name>__<param>` keys — see the caveat below.
- **Dedicated `Ensemble` category.** The four nodes use `category: "Ensemble"` (not
  `"Modeling"`) so they do **not** appear in the Basic/Advanced Training model pickers —
  they have their own canvas node instead.

### 🎨 Frontend — Dedicated Ensemble node

- **New `EnsembleNode`** (one node, in the Modeling palette group) — separate from Basic
  and Advanced Training. Styled with the same purple training-panel theme. The settings
  panel:
  - **Task** segmented toggle — *Classification* / *Regression*.
  - **Strategy** segmented toggle — *Voting* / *Stacking*. The two choices resolve to the
    matching core id (`voting_classifier`, `stacking_classifier`, `voting_regressor`,
    `stacking_regressor`).
  - **Training Mode** segmented toggle — *Basic* (with a collapsible **Base Model Hyperparameters** section to customize each chosen base model's fixed parameters individually) / *Advanced (Tuning)* (with tunable trial budgets, metric optimization target, search strategies, and a toggle to enable/disable nested base-learner tuning).
  - **Base Models** — add each model separately via a chip toggle list
    (`MultiSelectChips`), with a "pick at least two" guard.
  - **Voting type** (soft/hard) for classification voting; **Final estimator** + **Stacking
    CV folds** (with the nested-CV cost hint) for stacking.
  - **Target column**, a collapsible **Cross-Validation** section, and a **Start Ensemble
    Training** run button.
  - **Wired base learners (auto-detect).** Connect any Basic/Advanced Training model node
    to the ensemble's input and it is picked up automatically as a base learner — its
    `model_type` resolves to the ensemble base key and its hyperparameters seed the
    per-model params. Wired models override the in-node chip selection; a hint in the
    panel explains this.
  - **Inherited data source.** In the common `split → model → ensemble` flow (no direct
    data edge into the ensemble), the ensemble now inherits the dataset its base models
    consume, so the pipeline resolves a training dataset automatically. Base learners are
    spec-only, so the ensemble re-fits them on that single dataset.
  - **Cross-dataset wiring guard.** Connecting a model trained on a *different* dataset
    lineage into an ensemble now warns before committing the edge — catching accidental
    wiring across unrelated branches.
- **Shared wrapper hook `useTrainingNodeContext`** — encapsulates the upstream
  dataset-schema/target resolution and the run-pipeline action so the new node reuses the
  exact training plumbing **without modifying** the Basic/Advanced Training panels. (Those
  two files are intentionally left untouched; they can adopt the hook later.)
- **`pipelineConverter` — converter-side base-learner resolution (Phase 2).** The ensemble
  branch now separates model-spec edges from dataset edges by source node type: model nodes
  wired into the input become base-learner specs (translated to short ensemble keys via an
  inverted registry map; unsupported types like xgboost are skipped), are excluded from the
  data `inputs` so the backend never loads a model as a Dataset, and a model node wired
  *only* into ensembles is dropped from standalone training to avoid double-fitting. No
  backend/engine change was required.
- **`useGraphStore.onConnect`** suppresses the column-merge confirmation for `EnsembleNode`
  targets — a fan-in there is one dataset edge plus N model-spec edges, not a column merge.
- **Properties panel — no Merge Strategy for ensembles.** The "Merge Strategy" (last-wins /
  first-wins column resolution) dropdown is now hidden for `EnsembleNode`. Its multi-input
  fan-in is base-learner specs + one dataset, never a column merge, so the dropdown was
  misleading.
- **Canvas branches — base models fold into the ensemble branch.** `useBranchColors` now
  treats `EnsembleNode` as the real branch terminal and drops base-model nodes that feed
  *only* an ensemble from the branch list (same spec-only rule the converter uses for the
  backend). Result: a `data → model → ensemble` chain colours as **one** branch with a
  single `Path · Voting/Stacking` label instead of one extra Path tab per base model — so
  the canvas edge labels/colours and the Preview Results tabs stay aligned. An ensemble
  wired directly to data still earns its own branch tab.
- **Stacking `passthrough` toggle.** New Stacking-only boolean (Basic + Advanced modes,
  classifier + regressor) that lets the meta-learner also see the original input features
  alongside the base models' out-of-fold predictions. Threaded end-to-end: `_PASSTHROUGH_FIELD`
  in the hyperparameter specs, carried through `prepare_tuning_params` / `default_params` so
  tuning refits preserve it, surfaced as a checkbox in the Ensemble settings, and forwarded by
  the pipeline converter. Voting nodes silently ignore the flag (it is dropped before reaching
  the sklearn constructor).
- **Ensemble base models surfaced in Jobs + Experiments.** Job Details → *Tuning Configuration*
  now lists the ensemble's **Base Models** (plus Final Estimator / Passthrough / Voting for
  stacking) for advanced-tuning ensemble runs, and the Experiments → *Detailed Comparison* table
  gains a **Base Models** row (and a **Final Estimator** row when any compared run is a stacking
  model) so you can see which learners each ensemble combined side-by-side. Backed by a shared
  `extractEnsembleSummary` / `formatBaseEstimator` helper in `core/utils/format.ts` that reads
  the structural selection from nested hyperparameters (basic) or `tuning_config` (advanced).
- **Ensemble parallel jobs + Voting model weights.** New `n_jobs` control on every ensemble
  (fit base models in parallel; `-1` = all cores) and per-base-model **weights** for Voting
  models (classifier + regressor) so you can trust some learners more than others. Threaded
  end-to-end: `_N_JOBS_FIELD` in the hyperparameter specs, `n_jobs` coercion and `weights`
  length-validation in `_clean_meta_keys` (mismatched weights fall back to equal weighting,
  Stacking ignores weights), carried through `prepare_tuning_params` / `default_params`, surfaced
  as UI controls in the Ensemble settings, forwarded by the pipeline converter (weights sent as
  an estimator-aligned array only when non-default), and shown in Job Details → *Tuning
  Configuration*.
- **`MultiSelectChips` component** + `HyperparameterDef.type` extended with `'multiselect'`.
- **Nested parameter tuning support is fully featured**! You can easily toggle "Tune base model hyperparameters" in advanced mode.
- **Performance:** Stacking + an outer search = nested CV, which multiplies cost
  (outer folds × stacking `cv` × trials × base models). Keep the stacking `cv` small
  (e.g. 3) or use fewer trials when also running an outer search.
