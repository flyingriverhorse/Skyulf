# Changelog — 0.6.x Series

## v0.6.9

### 🐛 Bug Fixes — `skyulf-core` round 4 (`modeling`, more to follow)

Fourth audit pass via 2 parallel `explore` agents (`modeling`/`data` and `profiling`/`preprocessing`), prioritized by severity; verified with the full `skyulf-core` test suite (2655 passed), `ruff`, and `ty`. No frontend changes required for these two fixes - internal correctness only, no response-shape changes.

- **🔴 `StatefulEstimator.evaluate()` silently omitted test/validation splits for a Polars `SplitDataset`.** `has_test`/`has_val` and the `evaluate_split()` helper only recognized `pd.DataFrame` and `(X, y)` tuples - a bare Polars test/validation split fell through and was treated as absent, so `report["splits"]` only ever contained `"train"` with no error or warning. Fixed by adding an engine-agnostic `_is_non_empty_split()` helper and routing `evaluate_split()` through the same `_extract_xy()` used by `fit_predict()` (which already handled Polars correctly), instead of duplicating pandas-only extraction logic.
- **🔴 `perform_cross_validation()`'s `time_series_split` chronological sort (and time-column drop) was skipped entirely for Polars `X`.** The call site gated `_sort_by_time()` behind `isinstance(X, pd.DataFrame)`, so a Polars dataset's rows were never sorted before `TimeSeriesSplit` cut folds, and the time column was never dropped from features - leaking it directly into training. Verified: an out-of-order time column perfectly correlated with `y` produced `r2 == 1.0` under `time_series_split` CV with Polars input (vs. correct behavior with pandas). `_sort_by_time()` now sorts/drops for both engines (Polars via `.sort()` + a temporary joined column to keep `X`/`y` in lockstep, then split apart).
- **🔴 `DriftCalculator`'s PSI/KL divergence silently dropped out-of-reference-range values, hiding real drift.** `_calculate_psi`/`_calculate_kl` binned `expected`/`actual` (or `reference`/`current`) with `np.histogram(..., bins=breakpoints)`, which silently excludes any value outside `[breakpoints[0], breakpoints[-1]]` instead of counting it in a boundary bin. A column that fully shifted outside its reference range (e.g. all values above the historical max) could report **zero drift** because both histograms ended up empty. Fixed by `np.clip()`-ing both arrays into the breakpoint range before histogramming, so out-of-range values land in the first/last bin instead of vanishing. Added 2 regression tests asserting PSI/KL correctly flag a fully-shifted distribution as drifted.
- **🟠 `Casting` node crashed on the Polars engine when casting a string column (e.g. `"yes"`/`"no"`) to boolean.** Polars has no direct Utf8→Boolean cast at all — even `cast(pl.Boolean, strict=False)` raises `InvalidOperationError` — whereas the same pandas pipeline succeeds via a per-value alias table (`_coerce_boolean_value`). Added a Polars `pl.when/then` chain (`_bool_expr_from_string_col_polars`) that normalizes common true/false aliases (`yes/no`, `y/n`, `1/0`, `true/false`, `on/off`, etc., case-insensitive) before casting, matching pandas' alias table exactly. Unrecognized values become null under `coerce_on_error=True` (parity with pandas) and raise `ValueError` under `coerce_on_error=False` (parity with pandas' strict-mode re-raise). No frontend changes needed — internal engine-parity fix only.
- **🟠 `IterativeImputer` (MICE) ignored the user-configured `random_state`, always seeding sklearn with `0`.** `IterativeImputerCalculator.fit()` never read `random_state` from `config`, hardcoding `random_state=0` when constructing sklearn's `IterativeImputer`. Fixed to read `config.get("random_state", 0)`. **Frontend fix required and included**: `pipelineConverter.ts` also hardcoded `random_state: 0` when building the `IterativeImputer` step params, silently discarding whatever the UI's `random_state` field (`ImputationNode.tsx`) was set to — so the backend fix alone would have had no visible effect. Fixed the converter to forward `node.data.random_state`. Added regression tests on both sides (Python + a `pipelineConverter.test.ts` case).
- **🟡 Bucketing/binning nodes silently swallowed per-column exceptions with zero logging.** Both `_bucketing_apply_pandas()` (per-row binning of a column, e.g. degenerate edges) and `_fit_one_column_into_maps()` (per-column strategy fitting, e.g. `kmeans` on a non-numeric column) had bare `except Exception: continue`/`return` blocks — a column silently ended up unbinned with no trace in logs, making failures invisible in production. Added `logger.warning(..., exc_info=True)` before each `continue`/`return` so failures are now visible without changing behavior (still skip-and-continue, not raise). No frontend changes needed — internal observability improvement only.
- **🟢 `DataSplitter` didn't validate `test_size`/`validation_size` up front.** An invalid combination (e.g. `test_size + validation_size >= 1`, leaving zero rows for training) previously fell through to sklearn's `train_test_split` and surfaced a cryptic internal error. The frontend's Train/Test Split node already validates this client-side (`TrainTestSplitNode.tsx` shows "Error: Total split size exceeds 100%"), but the backend had no matching guard for API/programmatic callers. Added explicit `ValueError`s in `DataSplitter.__init__` for `test_size`/`validation_size` out of `[0, 1)` range and for their sum reaching `1`. No frontend changes needed — parity/defense-in-depth for the backend only.
- **🟢 Categorical profiling's `top_k` extraction assumed Polars' `value_counts()` struct always has exactly 2 keys.** `_analyze_categorical()` picked the "value" key by hard-coded position (`keys[0]` unless it was `"count"`, else `keys[1]`), which would silently misassign the value or raise `IndexError` if the struct ever carried an extra key. Not currently broken (Polars' struct is always 2-key today), but fragile. Fixed to find the non-`"count"` key by name instead of position, robust to any number of additional keys. No frontend changes needed — internal robustness fix only.
- **🟢 Logistic Regression's "newton-cholesky" solver was supported by the backend but missing from the UI dropdown.** `LogisticRegressionCalculator._SOLVER_PENALTIES` already validates `"newton-cholesky"` as a real supported solver (L2/None penalty only), but `LOGISTIC_REGRESSION_PARAMS`'s `solver` hyperparameter field — which the frontend dropdown renders from via the backend's hyperparameters API — never listed it as an option. Added `{"label": "Newton-Cholesky", "value": "newton-cholesky"}` to the options list. **Frontend automatically picks this up** — no frontend code change needed since the dropdown is generated dynamically from this backend registry.
- **🟢 `TuningApplier.predict()`'s legacy fallback (non-tuple `model_artifact`) crashed with an unclear `AttributeError` on Polars input.** The fallback unconditionally did `pd.Series(np.nan, index=df.index)`, but Polars `DataFrame` has no `.index` attribute. This path is genuinely reachable (an existing test pairs a plain calculator with `TuningApplier` and relies on the benign all-NaN placeholder before the artifact gets wrapped into a tuple), so it needed to keep working — not just raise. Made the placeholder engine-aware: uses `df.index` when available (pandas), otherwise builds a plain-integer-indexed all-NaN series sized to `len(df)` (Polars). Added a regression test with a Polars `DataFrame` input. No frontend changes needed — internal defensive-code fix only.

### 🐛 Bug Fixes — `skyulf-core` round 5 (`engines`, more to follow)

Fifth audit pass via 5 parallel `explore` agents across `core`/`data`/`engines`, `modeling`/`profiling`, `preprocessing`, and `backend`. Starting with the `engines` findings; verified with the full `skyulf-core` test suite (2678 passed), `ruff`, and `ty`. No frontend changes needed — these are all internal dataframe-engine-abstraction fixes with no API/response-shape changes.

- **🔴 Pandas `with_column()`/`__setitem__` silently corrupted data when assigning a `pd.Series` with a mismatched index.** Pandas aligns by index on assignment, while Polars always assigns positionally — so passing e.g. a re-indexed/filtered Series into `with_column("col", series)` produced an all-`NaN` column on pandas with no error, while the equivalent Polars call worked correctly. Added a `_to_positional_values()` helper that strips index alignment (converts to the target frame's index positionally) before assigning, and raises a clear `ValueError` if lengths genuinely mismatch instead of silently NaN-filling.
- **🟠 Polars `with_column()` crashed when given a bare scalar.** `pl.Series(name, values)` on a scalar creates a length-1 Series that fails to broadcast against a taller frame — unlike pandas' `assign()`, which broadcasts scalars automatically. Now uses `pl.lit(value).alias(name)` for non-array-like values so Polars broadcasts across all rows, matching pandas semantics.
- **🟡 `select()` had a str-vs-list engine parity gap.** Calling `select("col")` (a bare string instead of `["col"]`) on the pandas wrapper returned a raw `pd.Series` wrapped as if it were a DataFrame (breaking downstream `SkyulfDataFrame` assumptions), while Polars' `select("col")` correctly returns a one-column DataFrame. The pandas wrapper now normalizes a bare string to a single-element list before indexing, so both engines consistently return a DataFrame-backed wrapper.
- **🟢 `EngineRegistry` had no way to change the default/active engine at runtime.** Added `EngineRegistry.set_active_engine(name)`, validated against registered engines, complementing the existing `register()`/`get()`/`resolve()` API.

### 🐛 Bug Fixes — `skyulf-core` round 5 (`modeling`)

- **🔴 Hyperparameter tuning with `cv_type="time_series_split"` leaked the time column and evaluated folds out of chronological order.** Round 4 fixed this for `perform_cross_validation()`, but the parallel tuning path (`TuningCalculator.fit()`/`tune()`) converted `X`/`y` straight to numpy via `SklearnBridge.to_sklearn()` before ever sorting — discarding column names entirely, so the shared `_sort_by_time()` sort/drop logic never ran. Verified: an out-of-order time column perfectly correlated with `y` produced a near-perfect tuning CV score under `time_series_split`. Fixed by calling the same `_sort_by_time()` helper (already shared with `perform_cross_validation()`) before the numpy conversion whenever `cv_type == "time_series_split"`. Added a new `TuningConfig.cv_time_column` field so the time column can be specified (or auto-detected, same as regular CV). **No frontend changes needed** — the Advanced Tuning node's `cv_time_column` field and `pipelineConverter.ts` already forwarded this value in `tuning_config`; it was silently dropped on the backend because `TuningConfig` had no matching field to receive it. Added a regression test proving the leak is closed.

### 🐛 Bug Fixes — `skyulf-core` (`core`/`data`/`engines`/`modeling`/`profiling`)

Found via parallel `explore` agent audits of `core`, `data`, `engines`, `modeling`, and `profiling`, prioritized with a rubber-duck review; verified with the full `skyulf-core` test suite (2593 passed), `ruff`, and `ty` after each fix. No frontend changes required — all four fixes are internal to backend/core logic with no user-facing config surface.

- **Profiling `analyzer.py` silently recorded filters as "applied" even when the operator branch never matched** — the loop appended to `active_filters` unconditionally instead of only inside the matched operator branches, so an unrecognized operator or unknown column silently produced a no-op filter that was still reported as applied. Fixed to only record on an actual match, with warnings logged for unrecognized operators/unknown columns.
- **Tuning engine ignored `cv_shuffle`/`cv_random_state`** — all three search-phase CV splitter construction sites (`StratifiedKFold`/`KFold`/`ShuffleSplit`) hardcoded `shuffle=True` and reused the model's `random_state` regardless of the user's tuning config, making CV splits non-reproducible/non-configurable. Now honors `config.cv_shuffle`/`config.cv_random_state`, with a `random_state=None` guard when `shuffle=False` (sklearn raises otherwise).
- **Target-imbalance recommendation only ever considered the top 10 categories** — `_generate_recommendations` computed the imbalance ratio from a truncated `top_k` value distribution, so a severely underrepresented class outside the top 10 was invisible to the imbalance check. Added `_target_class_counts()` using the full `group_by` (falling back to `top_k` only past a 1000-unique-class safety cap).
- **`LogisticRegressionCalculator` could crash with a cryptic sklearn error on an incompatible solver/penalty combo** (e.g. `solver="liblinear"` + `penalty="elasticnet"`) — added a `fit()`-time validation seam that raises a clear `ValueError` up front. Only triggers when both keys are explicitly present in the same config, so partial overrides that fall back to a compatible default are unaffected.
- **`EDAVisualizer.plot()` leaked every matplotlib `Figure` it created** — none of the 7 `_plot_*` sections ever closed their figures after `plt.show()`, so repeated `plot()` calls accumulated open figures indefinitely. Now tracks which figure numbers were created during the call and closes only those, leaving any figures a caller already had open untouched.
- **Engine auto-detection matched by bare substring instead of the top-level module package** — `EngineRegistry.resolve()` used `"pandas" in module`/`"polars" in module`, which could misdetect an unrelated third-party module (e.g. `fake_polars_stub`) as the real engine. Now matches the top-level package exactly, while still correctly unwrapping Skyulf's own `SkyulfPandasWrapper`/`SkyulfPolarsWrapper` (guarded to the `skyulf.*` namespace, since polars' own `DataFrame` happens to have an internal `._df` too).
- **`SkyulfSchema.rename()` could silently produce duplicate column names** on a colliding rename mapping — now raises `ValueError` instead of corrupting the schema.
- **`InMemoryModelRegistry.register()` had a version-assignment race** under concurrent registration of the same model name — guarded with a lock so versions are always unique/contiguous.
- **Cross-validation `_aggregate_metrics` only used the first fold's metric keys** — a metric present in later folds but absent from fold 0 was silently dropped from the aggregated report. Now aggregates over the union of keys across all folds.
- **Regression evaluation's residual-plot downsampling was non-reproducible** — `evaluate_regression_model()` now accepts an optional `random_state` (default `42`) so repeated evaluations of the same model/data return the same sampled points.
- **Dead `pass` statement** removed from `PolarsEngine.to_arrow()`.
- **Geospatial analyzer accepted out-of-range coordinates** — a column merely named `lat`/`lon` holding unrelated numeric data (an ID, a percentage, etc.) was reported as valid geospatial bounds. Added a `[-90, 90]`/`[-180, 180]` range check.
- **Text PII alert claimed "Email/Phone" but only ever checked for email** — `_check_pii` now also applies a loose phone-number heuristic, matching what the alert message already claimed.
- **Ambiguous D/M/Y vs M/D/Y date-format disambiguation was silent** — when multiple candidate formats tied on the "most distinct months" heuristic, one was picked arbitrarily with no signal to the user. Now logs a warning listing the tied candidates.
- **Rule-discovery surrogate tree showed meaningless raw thresholds for categorical splits** (e.g. `"color <= 2.00"`, the internal ordinal-encoding split point) — rule text now renders `"color in ['red', 'blue']"` using the actual category labels. `RuleTree` gained an additive `categories` field (feature → ordered category labels) so the tree-graph visualization can do the same. **Frontend**: `RuleTreeGraph.tsx`/`RuleDiscoveryTab.tsx` updated to consume `categories` and render category-based split labels instead of raw thresholds.
- **Drift detection skipped all categorical/text columns entirely** — added PSI-based drift detection for them (category-frequency distribution, with a high-cardinality guard to skip free-text/ID-like columns and sample-size-scaled epsilon flooring for zero-proportion bins). **Frontend**: `DriftTable.tsx`/`useDriftReport.ts` updated to recognize the new `psi_categorical` metric (mapped to the same PSI column/threshold as numeric `psi`).
- **18 `print(f"Error ...")` exception-swallowing call sites across `profiling/_analyzer/*.py`** (column, dates, geo, multivariate, numeric, rules, target, temporal, text) replaced with proper module-level `logger.warning()` calls.

### 🐛 Bug Fixes — `skyulf-core` round 3 (more `modeling`/`profiling`/`core`/`engines`/`preprocessing`)

Third audit pass via 4 parallel `explore` agents (`modeling`, `profiling`, `core`/`engines`/`data`/top-level, and a fresh `preprocessing` re-audit), prioritized with a rubber-duck review; verified with the full `skyulf-core` test suite (2622 passed, 26 new regression tests), `ruff`, and `ty`. No frontend changes required — every fix is internal correctness/robustness with no new/changed API response shape surfaced distinctly in the UI.

- **🔴 `DateFeatures` silently produced all-null output on the Polars engine for ordinary date strings.** `pl.col(col).cast(pl.Datetime, strict=False)` only parses full ISO-8601 datetime strings — a plain `"2021-01-01"` (the default shape of a CSV date column) silently became `null` for every row, with no warning, while the pandas path parsed it correctly via `pd.to_datetime`. Existing parity tests always pre-converted the column to pandas datetime before building the Polars frame, masking the bug. Fixed to dispatch on dtype: string/Utf8 columns now go through `str.to_datetime(strict=False)`.
- **`pipeline.py` model-type resolution skipped its hardcoded fallback map on partial registry registration.** If `NodeRegistry.get_calculator()` succeeded but `get_applier()` raised (a partially-registered node), the fallback was gated only on `calculator is None`, so a working hardcoded applier was never tried — the pipeline raised a misleading "Unknown model type" instead. Fixed in both the main model-type branch and the `hyperparameter_tuner` base-model branch by resetting both on failure and gating on either being `None`.
- **Profiling `exclude_cols` didn't actually exclude columns from `missing_cells_percentage`, `duplicate_rows`, or `sample_data`.** All three were computed from the full unfiltered frame while only the reported column list was narrowed — excluding an all-null column inflated the missing-% (could read 100% instead of 0%), and excluded columns (e.g. PII) still leaked into `sample_data`. Fixed to compute all three against the narrowed column set.
- **Drift detection silently dropped `Int8`/`Int16`/`UInt*` columns entirely** — the numeric dtype allow-list only listed `Float32`/`Float64`/`Int32`/`Int64`, so narrower integer columns matched neither the numeric nor categorical branch and vanished from the report with no indication. Now uses polars' `dtype.is_numeric()`.
- **Grid/Random hyperparameter search silently fell back to untuned default params when every candidate failed on every fold** — `best_params` stayed `None`, coerced to `{}`, and tuning "completed" with `Best Score: -inf` and no error, unlike the halving/optuna strategies which already raise a clear `ValueError` for "no trials completed". Now raises the same error for consistency.
- **`EllipticEnvelope` outlier detector had no `random_state`**, unlike every other stochastic estimator in the codebase — repeated fits on identical data could produce different inlier/outlier decisions. Added a `random_state` config param (default `42`).
- **`H3Index` crashed the entire `.apply()` on a NaN/missing lat or lon** — unlike `GeoDistance`, which propagates `NaN` cleanly per row. Now catches bad/missing coordinates per-row and emits `None` for that row instead of aborting the whole column.
- **Outlier detection (`_detect_outliers`) sampled via `.head(limit)`** (first 50k rows) while PCA/clustering in the same analyzer used a random `.sample()` — for date/label-sorted data this systematically missed anomalies past the cutoff. Now uses the same seeded random sampling strategy.
- **`resolve_columns()` didn't deduplicate an explicit `columns` config list** — a duplicate column name (e.g. `["age", "age"]`) would be processed twice by stateful encoders/scalers, risking corrupted fitted artifacts. Now dedupes while preserving first-occurrence order.
- **Tuning engine's upfront NaN/Inf validation was skipped for object-dtype feature/target arrays** (mixed dtypes or leftover unencoded categorical columns) — the guard was gated on `np.issubdtype(dtype, np.number)`, so dirty data in an object array reached the model and produced an opaque downstream sklearn error instead of the intended clear message. Now also scans object arrays via `pd.isna`.
- **`NodeRegistry.register()` (top-level node registry) had no thread lock**, unlike `InMemoryModelRegistry` (hardened in round 2) — concurrent registration of the same node name could interleave a calculator from one call with an applier from another, directly enabling the partial-registration bug above. Added the same lock pattern.
- **Binary classification precision/recall/F1 silently omitted for non-`{0,1}` labels** — `calculate_classification_metrics()` relied on sklearn's `pos_label=1` default, so string labels (e.g. `"yes"/"no"`) or `{-1, 1}` integer labels never matched and the metrics were quietly dropped from the report. Now resolves the actual positive-class label from `model.classes_[1]`, matching the existing pattern already used in `modeling/_evaluation/classification.py`.
- **Multiclass ROC-AUC/PR-AUC silently vanished from a CV fold's metrics whenever that fold's y_true happened not to include every class** — `roc_auc_score`/`label_binarize` were called without an explicit `labels=classes` argument, so a fold missing a rare class produced a shape mismatch that was caught and swallowed rather than computed against the full label set. Now passes `labels=classes` explicitly so all folds report these metrics consistently.
- **`pack_pipeline_output()` reattached `y` to `X` via a naive `pd.concat([X, y], axis=1)`** with no check that the two shared a compatible index — a preprocessing step that reset/changed one but not the other could silently NaN-pad or duplicate rows instead of raising. Now raises `ValueError` on an outright row-count mismatch (always a real bug), and resets both to a shared positional index before concatenating when counts match but indices merely differ (the common, benign case).
- **`SkyulfSchema`'s dtype-extraction helpers swallowed every exception silently** (`_extract_pandas_dtypes`/`_extract_polars_dtypes`, `bare except Exception: return {}`) — now log at `debug` level so a genuinely broken frame's schema-inference failure is at least visible in diagnostics instead of vanishing.
- **`EngineRegistry.resolve()` logged a spurious "Unknown data type" warning for plain Python lists/tuples** (a common, expected shape for a raw `y` target) — now only warns for types that are genuinely unrecognized.
- **`SplitDataset.copy()` silently returned the same object reference** for any payload with neither `.copy()` nor `.clone()` (e.g. a plain list/dict), defeating copy semantics for that slot. Now falls back to `copy.copy()` so the result is always independent of the original.
- **`SklearnApplier.predict_proba()` used raw numpy `classes_` values as DataFrame columns**, risking JSON-serialization issues downstream for numpy scalar types (e.g. `np.int64`). Now coerces to `str`, mirroring the existing `class_names` normalization in `modeling/_evaluation/classification.py`.
- **`CalibratedClassifierCV`'s `cv` parameter was silently coerced from an invalid value (0 or 1) to 2** with no signal — now logs a warning when the coercion actually happens.
- **Feature-selection's classification/regression inference cutoff (`<= 10` distinct values → classification) was an undocumented magic number** — added a named constant, a debug log noting when the heuristic fires, and boundary tests at exactly 10 vs. 11 distinct values.
- **Nested CV's docstring/comments claimed the inner loop "selects best hyperparameters"** — it actually just refits the same config across inner sub-splits as a diagnostic stability signal; no hyperparameter search happens. Clarified the docstring and inline comments (implementing real inner-loop search deemed out of scope for this pass).
- **Nested CV recorded an inner-fold fit failure as a score of `0.0`** — for regression, `0.0` is itself a meaningful R² value, so a hard failure silently corrupted `inner_cv_mean` instead of being excluded. Now uses `NaN` internally (filtered from the mean) and surfaces `inner_cv_mean` as `None` when every inner fold in an outer fold failed; failures are now also surfaced via `log_callback`, not just `logger.warning`.
- **`perform_cross_validation()` duplicated `_build_splitter()`'s branching logic inline**, and the duplicate had no warning for an unrecognized `cv_type` (silently falling back to `KFold`), unlike `_build_splitter()` (used only by nested CV) which already warned. Deduplicated by having `perform_cross_validation()` call `_build_splitter()` directly, so both paths now share the same warning behavior.
- **Profiling's "Possible ID" `is_unique` recommendation was dead code for Text/Numeric columns** — it was only ever computed inside the Categorical semantic-type branch, so a fully-unique numeric or text ID-like column (e.g. a UUID stored as text) was never flagged. Now computed generically for every column from the already-collected `__unique` stat, using the same `>50`-unique threshold as the Categorical branch (avoids noise on small tables).
- **Boolean-semantic-type target columns were skipped entirely by target-relationship analysis** — `analyzer.py` only handled Numeric/Categorical targets, so a boolean target (e.g. `is_churned`) got no per-feature correlation/relationship output at all. Extended to also run for Boolean targets; fixed a related crash this surfaced — polars raises `InvalidOperationError` casting `Boolean` directly to `Categorical`, so the target is now cast to `Utf8` first.
- **Univariate/model-based feature selection silently `fillna(0)`'d missing values before scoring**, which can bias a feature's score toward/away from significance depending on its distribution, with no way to know it happened. Added a shared `_fillna_zero_with_warning()` helper (now used by both `univariate.py` and `model_based.py`) that logs a warning naming the affected columns.
- **Correlation computation silently truncated to the first 20 numeric columns** with no indication that any columns were dropped from the analysis. Now logs a warning naming the excluded columns when truncation occurs.

### 🔒 Deep Security & Reliability Hardening — Part III

Third-pass audit across the database layer, ML pipeline routers, execution services,
infrastructure/middleware, EDA router, monitoring tasks, and model registry API.
28 issues found and fixed.

#### High Fixes

- **Cursor closed before fetch** (`backend/database/data_sources/async_postgres_queries.py`). `fetchone()` was called after `commit()` — asyncpg closes the server-side cursor on commit, silently returning `None` for every `INSERT … RETURNING` statement. `fetchone()` is now called before `commit()`.
- **Double-commit on `engine.begin()` block** (`backend/database/async_connection_manager.py`). An explicit `await conn.commit()` inside an `engine.begin()` context manager caused an `InvalidRequestError` — `engine.begin()` already commits on clean exit. Explicit commit removed; same block also had a double-rollback on the exception path which is now cleaned up.
- **Empty initialisation stub** (`backend/database/async_init_db.py`). `initialize()` created settings then fell off the end, always returning `None` — the database was never initialised on startup. Routing logic added to call the correct `_try_init_*` helper based on `DATABASE_URL`.
- **Path traversal in `safe_delete_path`** (`backend/utils/file_utils.py`). Accepted any absolute path with no boundary check — callers passing DB-derived paths could delete arbitrary files. Added `_is_within_base()` helper and enforced `allowed_base` containment.
- **Path traversal via `source_name` fallback** (`backend/utils/file_utils.py`). A user-controlled DB label was used as a filesystem path fallback; a crafted value (`../../sensitive`) could return arbitrary paths. The `source_name` path fallback has been removed entirely.
- **Error message leak in `meta.py`** (`backend/ml_pipeline/_internal/_routers/meta.py`). Raw Python exception text (file paths, connection strings) was interpolated into `SkyulfException` messages and returned in HTTP responses via `str(e)`. Replaced with a generic message; full detail retained in server-side `logger.exception`.
- **Session resource leak in `preview.py`** (`backend/ml_pipeline/_internal/_routers/preview.py`). `sync_session` was created before the inner `try/finally` block — if `partition_parallel_pipeline()` raised between creation and the inner `try`, the session was never closed. Initialised to `None` before the outer `try`; closed unconditionally in the outer `finally`.
- **Unvalidated `operator` field in EDA `FilterRequest`** (`backend/eda/router.py`). Any arbitrary string was forwarded to the EDA analyzer's filter logic, enabling abuse/injection. Added `field_validator` restricting to the valid set: `{==, !=, >, >=, <, <=, in, not_in, contains, not_contains}`.

Raw `ValueError` message (potentially containing internal paths or DB details) was returned directly to the HTTP client via `detail=str(e)`. Now logged server-side; generic `"Job artifacts not found"` returned to client.

- **Deprecated `asyncio.get_event_loop()`** (`backend/database/adapter.py`). Raises `DeprecationWarning` in Python ≥ 3.10 and `RuntimeError` in ≥ 3.12 inside an already-running event loop. All 5 occurrences replaced with `asyncio.get_running_loop()`.

When `_bundle_transformers()` failed, only `logger.error()` was called — no `raise`. The pipeline would proceed, mark the job as succeeded, and produce an artifact that couldn't be deployed. Fixed to `logger.exception(...); raise`.
- **Warning capture handler swallowed formatting errors** (`backend/ml_pipeline/_execution/engine/_warning_capture.py`). Errors during log-record formatting/emission were silently discarded. Now calls `self.handleError(record)` per the `logging.Handler` contract and logs a debug message.
- **`start_time` schema default not UTC-aware** (`backend/ml_pipeline/_execution/schemas.py`). `default_factory=datetime.now` produced a naive local timestamp. Changed to `default_factory=lambda: datetime.now(timezone.utc)`.
- **`list_jobs` pagination OOM cap** (`backend/ml_pipeline/_execution/jobs.py`). The cross-table merge-sort fetches `skip + limit` rows from each table in Python. With no upper bound, a large `skip` value could allocate hundreds of MB. A `_MAX_SKIP = 500` sentinel now clamps the offset and emits a warning log for out-of-range requests.

### 🔒 Security & Reliability Hardening

#### Critical Fixes
- **Path traversal blocked** (`backend/data/catalog.py`). `FileSystemCatalog._get_path` now enforces base-directory containment for **absolute paths** — previously only `..` segments were blocked, allowing any absolute path outside the upload directory to be read via the pipeline preview/run endpoints. All paths (absolute and relative) must now resolve inside `UPLOAD_DIR`.
- **Internal error messages no longer leaked** (`backend/main.py`). The global `general_exception_handler` previously returned `"Internal server error: {real_message}"` exposing stack details, file paths, and DB errors to HTTP clients. Now returns a generic `"Internal server error"` body while full detail is retained in logs and Sentry.
- **S3 / datasource credentials redacted from API responses** (`backend/data_ingestion/schemas/ingestion.py`). `DataSourceRead.config` now strips sensitive keys (`storage_options`, `aws_secret_access_key`, `aws_access_key_id`, `password`, `api_key`, `token`, etc.) before serialisation.

#### High Fixes
- **PostgreSQL migration transaction isolation** (`backend/database/engine.py`). Each DDL in `_run_migrations()` now runs in its own `async with engine.begin()` transaction. Previously all migrations shared one transaction — a "column already exists" error on PostgreSQL aborted the entire batch and crashed startup.
- **`deployment/predict` HTTPException masking fixed** (`backend/ml_pipeline/deployment/api.py`). A broad `except Exception` was catching `HTTPException` and converting intended 404s (no active deployment) into 500s. Added `except HTTPException: raise` guard; also removed `str(e)` from the 500 detail to prevent error leakage.
- **Double error recording eliminated** (`backend/main.py`). Unhandled exceptions were recorded twice — once in `general_exception_handler` and again when forwarded to `generic_http_exception_handler`. Handler now returns directly, recording once.
- **Cross-worker job submission race mitigated** (`backend/ml_pipeline/_execution/jobs.py`). `find_active_job` now uses `.with_for_update(skip_locked=True)` so the idempotency check is serialised at the database level under PostgreSQL multi-worker deployments, not just within a single process's event loop.
- **Stale job reset no longer corrupts live Celery jobs** (`backend/main.py`). `_reset_stale_jobs()` previously failed all `running`/`queued` jobs on every API restart, including jobs actively running in Celery workers. A 2-hour `started_at` cutoff now protects recently-started jobs.

#### Medium Fixes
- **Rate limiting added to expensive endpoints**. Upload `10/min`, drift calculate `20/min`, deploy `10/min`, predict `60/min` — all keyed by client IP via `slowapi`.
- **Drift upload size enforced** (`backend/monitoring/router.py`). `POST /monitoring/drift/calculate` previously read the entire uploaded file into memory with no cap. Now enforces `MAX_UPLOAD_SIZE` and returns HTTP 413 if exceeded.
- **`_submit_locks` memory leak fixed** (`backend/ml_pipeline/_internal/_routers/run_pipeline.py`). The process-global dict of asyncio locks grew unboundedly with unique pipeline keys. Locks are now evicted after each submission via `_release_submit_lock()`.
- **UTC timestamp fix** (`backend/monitoring/router.py`). The `slow-nodes` endpoint computed its time window cutoff with `datetime.now()` (local time) while timestamps are stored as UTC — causing incorrect filtering on servers with a non-UTC offset. Fixed to `datetime.now(timezone.utc)`.

#### Configuration & Defaults
- **`SECRET_KEY` required in production** (`backend/config/base.py`). A new `model_validator` raises on startup if `FASTAPI_ENV=production` and `SECRET_KEY` was not explicitly set. The old default (`secrets.token_urlsafe(32)`) generated a new random key each restart, breaking JWT validation across workers and after restarts.
- **Hardcoded `AUTH_FALLBACK_PASSWORD` removed** (`backend/config/mixins/security.py`). Default changed from `"admin123"` to `""`. A validator now rejects an empty password if `AUTH_FALLBACK_ENABLED=true`, requiring explicit configuration.
- **`extra="ignore"` in Settings** (`backend/config/base.py`). Changed from `extra="allow"` — unknown environment variables are no longer stored as accessible attributes on the settings object.
- **Dead `SESSION_COOKIE_*` settings removed** (`backend/config/mixins/security.py`, `environments.py`). No `SessionMiddleware` is wired in the application; these settings created a false impression of active cookie security. A comment marks the fields as reserved for future use.
- **Duplicate ML router mounts removed** (`backend/main.py`). The ML pipeline router was mounted at three paths (`/api/pipeline`, `/ml-workflow/api/pipelines`, `/api/ml`). The two legacy aliases were not referenced by any frontend code and have been removed. Canonical path remains `/api/pipeline`.

## v0.6.0 — Ensemble models (Voting / Stacking)

First release of the **ensemble meta-model** family. Combine several base
learners into one estimator, for both classification and regression, directly
from the existing training node.

### 🆕 skyulf-core — Ensemble nodes

- **New `modeling/ensemble.py`** with four registered nodes (Calculator + Applier
  each), kept in their own module separate from `classification.py` / `regression.py`:

  | Registry id | sklearn class | Task |
  |---|---|---|
  | `voting_classifier` | `VotingClassifier` | classification |
  | `stacking_classifier` | `StackingClassifier` | classification |
  | `voting_regressor` | `VotingRegressor` | regression |
  | `stacking_regressor` | `StackingRegressor` | regression |

- **Advanced tuning & per-model hyperparameters in ensembles:**
  - Ensembles now support **Basic mode with fixed hyperparameters** per base model (e.g., custom `n_estimators` for a random forest) and the final estimator (for stacking).
  - Ensembles now support **Advanced/Tuning mode**, running through hyperparameter optimization with search strategies like Random, Grid, Optuna, Halving Random, or Halving Grid.
  - Added support for **nested parameter tuning** inside Advanced mode, auto-generating search spaces for all chosen base estimators (e.g. `random_forest__n_estimators`) and the stacking final estimator.
- **Selectable base learners.** The frontend sends `base_estimators` as a list of
  string keys (e.g. `["random_forest", "logistic_regression"]`); a shared resolver
  (`_BaseEnsembleCalculator`) turns them into `estimators=[(name, instance), …]` —
  mirroring the `base_estimator` pattern in `CalibratedClassifierCalculator`. Unknown
  keys are skipped with a warning; an empty selection falls back to sensible defaults
  so the node always trains. Stacking additionally resolves a `final_estimator` key.
- **Classification base models:** logistic regression, random forest, extra trees,
  gradient boosting, hist gradient boosting, AdaBoost, decision tree, Gaussian NB,
  SVC (probability-enabled), KNN — plus XGBoost / LightGBM when those optional wheels
  are installed (auto-detected only, omitted from the manual picker).
  **Regression base models:** linear regression, ridge, lasso, elasticnet, random
  forest, extra trees, gradient boosting, hist gradient boosting, AdaBoost, decision
  tree, SVR, KNN — plus XGBoost / LightGBM when available.
- **Cross-validation semantics** (the important bit):
  - **Voting** does *no* internal CV — each base model is fit once, then predictions
    are averaged (soft = mean proba, hard = majority vote) / averaged (regression).
  - **Stacking** uses an internal `cv` (default 5) to generate the base models'
    out-of-fold predictions for the final estimator. This is *required* — without it
    the meta-learner trains on in-sample predictions and over-fits. Same shape as
    `CalibratedClassifierCV`'s internal `cv`.
- **Tests:** `skyulf-core/tests/test_ensemble.py` (8 tests) — registration, soft-proba
  range, hard-vote labels, stacking fit with `cv`, regression predictions, unknown-key
  skip, and empty-selection fallback.

  pops the transport keys in `_resolve_estimators` and `_maybe_calibrate` wraps each base learner

  `prepare_tuning_params`, exposed as hyperparameter fields, surfaced in the Ensemble settings
  (classification-only section), forwarded by the pipeline converter, and shown in Job Details →
  *Tuning Configuration*.
- **Calibrated base-model tuning crash fixed.** Tuning a classification ensemble with calibration
  enabled raised `ValueError: Invalid parameter 'n_estimators' for estimator
  CalibratedClassifierCV` — nested search-space keys addressed the wrapper, not the wrapped model.
  `build_ensemble_search_space` now emits `<name>__estimator__<param>` when calibration is on so
  tuning and the post-tuning refit route base params correctly (final estimator unchanged).
- **Manual task override no longer locked by wired models.** Picking the Task (Classification /
  Regression) by hand now sticks even when a model node is wired in — the auto-sync effect honours
  `task_manual` instead of forcing the task back from the connected model's type.
- **Two-column expanded layout.** The Ensemble settings split into two side-by-side columns when
  the node is opened wide (primary setup on the left; strategy / tuning / calibration / CV on the
  right); the narrow sidebar stays single-column.
- **`MultiSelectChips` component** + `HyperparameterDef.type` extended with `'multiselect'`.
- **Nested parameter tuning support is fully featured**! You can easily toggle "Tune base model hyperparameters" in advanced mode.
- **Performance:** Stacking + an outer search = nested CV, which multiplies cost
  (outer folds × stacking `cv` × trials × base models). Keep the stacking `cv` small
  (e.g. 3) or use fewer trials when also running an outer search.
