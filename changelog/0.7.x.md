# Changelog — 0.7.x Series

## v0.7.2


**`skyulf-core` usability improvements** (from a dedicated API-discoverability review): `EDAAnalyzer`, `EDAVisualizer`, `DatasetProfile`, `DriftCalculator`, and `expect_*` validators are now re-exported directly from the top-level `skyulf` package (previously only reachable via `skyulf.profiling`, despite being the library's headline EDA feature); the evaluation/SHAP-explanation helpers previously only reachable through the underscore-prefixed `skyulf.modeling._evaluation`/`_explainability` packages are now re-exported from `skyulf.modeling`; and `NodeRegistry` (with two new discovery helpers, `list_transformers()`/`list_models()`, thin wrappers over the existing `get_all_metadata()`) is now exported top-level too, so a caller can list every valid transformer/model name without reading source. `SkyulfPipeline.predict()` now raises a clear, actionable `ValueError` (naming the offending column) if the input still contains the target column used during `fit()`, instead of an opaque raw sklearn feature-count `ValueError` several layers down. Unknown-transformer/unknown-model config errors (`FeatureEngineer`, `SkyulfPipeline`) now forward the registry's own "Available: [...]" hint instead of discarding it, so a config typo is actually correctable from the error message alone. The `viz`-extra optional-dependency messages in `EDAVisualizer` now point at `pip install skyulf-core[viz]` instead of individual raw package names (`matplotlib`, `rich`), matching what the extra is actually named in `setup.py`. No behavior change for any existing caller — all changes are additive exports/messaging.

**Lint fixes**: `skyulf-core/skyulf/preprocessing/pipeline.py`'s `isinstance` check now uses `X | Y` union syntax instead of a tuple (Ruff `UP038`); `backend/ml_pipeline/_execution/engine/_node_runners.py` dropped an unused `StepType` import (Ruff `F401`, it was only referenced inside a docstring).

**More `skyulf-core` usability fixes** (same API-discoverability review, follow-up batch): the `"Split"` node alias now emits a `DeprecationWarning` pointing callers at the canonical `"TrainTestSplitter"` name instead of silently supporting two names for the same node forever. A new opt-in static check, `skyulf.validate_leakage_safety(config)` / `SkyulfPipeline(config).validate_leakage_safety()`, walks a pipeline config and returns human-readable warnings if a data-dependent preprocessing step (imputer, scaler, encoder, feature selector, outlier detector, etc.) is wired upstream of `TrainTestSplitter` — the same leakage pattern the existing hard-block pre-execution check catches, now also checkable before ever building/fitting the pipeline. Parameter-validation error messages now list the valid choices on the Polars-only path that was missing them (`SimpleImputer` strategy, `GeoDistance` method/unit, `FeatureInteraction` degree, tuning strategy, problem type) via a new shared `raise_invalid_choice` helper — the Pandas path already got this via sklearn's own validation, so no change was needed there; the preprocessing dispatcher also now logs expected user-config `ValueError`s at DEBUG without a stack trace instead of `logger.exception`, reserving full tracebacks for genuinely unexpected exceptions. `TuningCalculator`, `SimpleImputerCalculator`/`Applier`, `SkyulfPipeline`, and `FeatureEngineer` gained missing docstrings, and the `node_meta` decorator now backfills any node class's empty/`None` `__doc__` from its registered metadata. `SkyulfPipeline`/`FeatureEngineer` construction now runs a conservative, structural-only Pydantic validation pass over the config shape (e.g. catches a typo'd `"transformer"` key with a "did you mean" hint) — per-node `params` internals remain fully unrestricted (`extra="allow"`), and no previously-valid config is rejected; verified against the full `skyulf-core` and backend suites with no regressions.


## v0.7.1

**`skyulf-core` example notebooks now showcase every previously-unused preprocessing category.** An audit of the full `skyulf/preprocessing/` module tree against all 9 example notebooks (`examples/00_quickstart.ipynb` through `08_online_retail_customer_segmentation.ipynb`) found five feature families that had never been demonstrated anywhere: outlier handling, feature selection (beyond the existing variance/correlation filters), feature generation, time-series feature engineering, and geo features. All five now have a real, executed showcase:

- **Outlier handling** (`01_house_prices_regression.ipynb`): a new section fits the same untuned Ridge model three ways — no outlier handling, `Winsorize` capping, and `IQR`-based row removal — on identical splits so the comparison is apples-to-apples. IQR row-removal is the genuine winner on this dataset (R² 0.9015 → 0.9191, log-RMSE 0.1356 → 0.1208).
- **Feature selection strategies** (`05_santander_imbalanced_classification.ipynb`): a new section compares the existing `VarianceThreshold`/`CorrelationThreshold` baseline against `UnivariateSelection` (ANOVA F-test) and `ModelBasedSelection` (Random Forest importance), on the same 369-column wide frame — ROC-AUC differs meaningfully across the three (0.7882 / 0.7750 / 0.7681).
- **Feature generation** (`07_spaceship_titanic_classification.ipynb`): a new section runs `FeatureInteraction` and `PolynomialFeatures` on spend/age columns against the existing baseline Logistic Regression. Reported honestly: the plain baseline actually beat both generated-feature variants here (F1 0.7914 vs. 0.7749 vs. 0.7007) — not every technique wins on every dataset, and the notebook says so. The same notebook also gained a Grid-Search-tuned Random Forest section alongside the existing Random-Search-tuned one.
- **Time-series features** (`08_online_retail_customer_segmentation.ipynb`): a new bonus section builds a monthly revenue series from raw transaction dates and demonstrates `DateFeatures`, `LagFeatures`, and `RollingAggregate` via `FeatureEngineer(...).fit_transform()` (a preprocessing-only run, no modeling step) — explicitly noted as a feature-engineering demo rather than a forecasting model, since ~13 months of history isn't enough data for one.
- **Geo features** (`00_quickstart.ipynb`): a new section attaches synthetic per-city jittered lat/lon coordinates plus a fixed warehouse location, then demonstrates `GeoDistance` (haversine distance to warehouse — correctly ~5km for London customers vs. ~6,800km/~10,800km for Nairobi/Singapore) and `H3Index` (hex-grid spatial index, correctly clustering by city). Also shows `EDAAnalyzer(..., lat_col=, lon_col=)`'s built-in geospatial profiling, rendered as a standalone scatter chart built from the public `geospatial.sample_points` data (kept separate from `EDAVisualizer.plot()`, which renders every EDA chart at once and would otherwise be noisy for a single-purpose demo). Requires the existing `skyulf-core[geo]` extra (`h3` package) for the index portion.

**Three real library bugs were found and fixed** while building/executing these notebooks against real data (not notebook-only workarounds):

1. `feature_selection/_common.py`: `_extract_target()` returned a raw Polars `Series` for `y` instead of converting it to pandas, causing `_infer_problem_type()` (which assumes pandas' `.empty`/`.dropna()`) to crash with `AttributeError: 'Series' object has no attribute 'empty'` on the Polars engine. Fixed by converting in `_extract_target()` itself.
2. `feature_selection/_common.py`: `UnivariateSelection`'s own documented default, `method: "SelectKBest"` (PascalCase, from its `@node_meta` params), silently no-opped — the internal builder dict only recognized snake_case keys (`"select_k_best"`), so the node's *default configuration* dropped zero columns instead of the requested `k`. Fixed with a new method-name alias/normalization layer so both spellings work.
3. `profiling/visualizer.py`: `EDAVisualizer._plot_scatter_matrix` passed a full-length target-color array into pandas' `scatter_matrix`, but pandas internally NaN-filters each column pair's x/y values without filtering the color array to match — crashing with a shape-mismatch `ValueError` whenever any plotted numeric column had missing values (a very common case). Fixed by dropping NaN rows across the selected columns up front so colors and coordinates always stay aligned.

All three fixes were verified against their existing test suites with no regressions (124 feature-selection tests, 34 visualizer tests, all passing), plus full notebook re-execution.

**Docs:** `skyulf-core/README.md` and `skyulf-core/examples/README.md` example tables updated to describe all five new sections; the quickstart notebook's "Next steps" list also gained a missing link to `08_online_retail_customer_segmentation.ipynb`. Final verification pass confirms all 9 example notebooks execute end-to-end with zero errors and no `pandas` import anywhere (Polars/NumPy only, per the library's no-hidden-pandas design goal).

**Data leakage hard-block:** pipelines are user-built DAGs with no enforced node order, so a data-dependent preprocessing node (Imputer, Scaler, Encoder, feature selector, outlier detector, etc.) could previously be wired *upstream* of a `TrainTestSplitter`, silently fitting its statistics on the whole dataset — train and test combined — before the split happened. A new pre-execution validation pass (`backend/ml_pipeline/_execution/_leakage_validation.py`) now walks the pipeline graph and hard-blocks execution with a clear, actionable error if this ordering is detected, before any node runs; it's a no-op for inference-only pipelines with no splitter, and correctly ignores `FeatureTargetSplitter` (which only separates X/y and creates no train/test boundary). The same check now also runs client-side on the canvas (`pipelineLeakageValidation.ts`) so users get instant feedback instead of waiting for a job to fail server-side. The repo's own `test_full_inference_pipeline.py` reference pipeline — which had been demonstrating exactly this leaky order — was corrected to the safe ordering.

**Solver/penalty validation gap fixed:** `LogisticRegressionCalculator`'s solver/penalty compatibility check previously only inspected the raw config override dict, so overriding just `penalty` (very common, since `solver` defaults to `lbfgs`) silently skipped validation and let invalid combinations reach sklearn's own opaque `ValueError` at fit time. The check now validates the merged default+override params, both for plain fits and for ensemble base/final estimators (`VotingClassifier`/`StackingClassifier`), which previously bypassed validation entirely via `set_params()`.

**sklearn warning cleanup:** `LogisticRegression`'s deprecated `penalty` param is now consistently normalized to `l1_ratio`/`C` across all three instantiation paths (plain fit, ensemble, hyperparameter tuning) — previously tuning bypassed the existing normalization helper, so live sessions saw a `FutureWarning` on every tuned logistic-regression run. `ConvergenceWarning` (raised via Python's `warnings` module, invisible to the UI) is now bridged into the existing `logger.warning()`-based job-log surface for every sklearn model (base fit and tuning search/refit), so users see convergence issues in the job log instead of only in server stderr. `max_iter` ceilings were also raised for `logistic_regression` (saga) and `ridge_regression` (sag/saga) hyperparameter search spaces to reduce how often the warning fires in the first place.

**Stale dataset 404 handling:** the frontend now distinguishes a genuine 404 (stale/deleted dataset reference) from other errors when fetching a dataset's schema — showing a friendlier "dataset not found" message instead of retrying, and no longer retries at all on 4xx responses (only 5xx), cutting out wasted retry spam in the browser console/network tab.

Fully removes the legacy `BASIC_TRAINING`/`ADVANCED_TUNING` `StepType` enum members and every old hidden canvas node (`BasicTrainingNode`, `AdvancedTuningNode`, generic `TrainingNode`) that were kept since the original training-unification work for backward compatibility with pre-unification saved canvases. Every training-family node — Classification, Regression, Text Classification, Segmentation, and all Ensemble variants — now exclusively emits the canonical `training` step_type + `run_mode` param throughout the backend and frontend, with no legacy branches remaining anywhere in live source.

The `job_type` API/dispatch field used to distinguish fixed-run from tuned-run has been renamed from `"basic_training"`/`"advanced_tuning"` to `"training"`/`"tuning"` for naming consistency with the unified `TrainingJob` schema, with no behavior change.

Also fixes a real, previously-existing bug as a side effect of this cleanup: the execution-mode (merge/parallel) toggle, canvas branch coloring, connection-type validation, and perf-overlay bucketing previously only recognized the old legacy node `definitionType` keys (`basic_training`, `advanced_tuning`, `model_training`, `hyperparameter_tuning`) and silently didn't apply to the newer canonical Classification/Regression/Text Classification/Training node types. All four features now correctly recognize the canonical types.

**Dev-only breaking change:** pre-existing saved canvases and job history using the old legacy node types or `job_type` values (`basic_training`/`advanced_tuning`) are no longer supported and must be recreated. This is intentional — the dev database was wiped and recreated as part of this cleanup.

Internal refactor: extracted the duplicated cancel/log-append/status-update logic shared by `BasicTrainingManager` and `AdvancedTuningManager` into a common `TrainingJobManagerBase` helper. No behavior, API, or DB schema changes — first step of unifying Basic Training and Advanced Tuning into a single flow.

Continues the unification with the orchestration and execution layers: `StepType` gains a single canonical `training` value (with `basic_training`/`advanced_tuning` retained as backward-compatible aliases so already-saved pipeline configs and job rows keep executing unchanged), and the pipeline engine now dispatches every training node through one path driven by a `run_mode` param (`"fixed"` for a single configured model, `"tuned"` for a multi-candidate hyperparameter search) instead of two separate step handlers. `run_mode="fixed"` reuses the exact same tuning-engine machinery as `"tuned"`, just degenerated to a single trial, so both flows now share one code path end-to-end from node config to trained model.

Rounds out the frontend to match: a single generic `TrainingNode`/`TrainingSettings` component (keyed by `run_mode: 'basic' | 'advanced'`) replaces the separate Basic Training / Advanced Tuning settings UIs, reusing the same hyperparameter, search-space, and strategy widgets either way. Four task-scoped node wrappers — **Classification**, **Regression**, **Text Classification**, and **Segmentation** — now appear in the node palette (derived from each model's registry tags), replacing the old generic Basic Training/Advanced Tuning node choice; the legacy nodes are kept registered but hidden so previously-saved canvases keep loading with zero migration. The Experiments page's job-history filters and comparison view were similarly updated to filter by task type instead of basic/advanced, including a fix so all four clustering algorithms (not just K-Means) correctly surface under the Segmentation tab.

Finishes the unification on the database side: `BasicTrainingJob` and `AdvancedTuningJob` — previously two parallel tables with duplicated columns — are replaced by a single `TrainingJob` table discriminated by `run_mode` (`"fixed"`/`"tuned"`), with `run_number`/`version` unified into one `version` column. Every job-facing code path (job service, `/jobs` and `/stats` endpoints, model registry version counting, monitoring drift/slow-node scans) now reads from the one table scoped by `run_mode` instead of querying two tables and merging results. This is a schema-only change with no data migration — the dev database was dropped and recreated rather than backfilling old rows — so it only affects fresh environments; `BasicTrainingManager`/`AdvancedTuningManager` remain separate manager/strategy classes for now (planned for later cleanup once this is stable).

Legacy Basic Training and Advanced Tuning canvas nodes now dispatch through the same shared `training`/`run_mode` execution path as the new unified Classification/Regression/Text Classification/Segmentation nodes (previously only newly-created nodes emitted the canonical `step_type`; legacy nodes kept emitting `basic_training`/`hyperparameter_tuning`). Effective run mode for a legacy node is derived from its own `run_mode` field when present, falling back to a `definitionType`-based inference otherwise, so already-saved canvases keep executing with identical behavior with no migration needed. Fixed a related pre-existing bug (introduced by the earlier training-methods refactor) where fixed-mode ensemble training (Voting/Stacking Classifier/Regressor) crashed with `VotingClassifier.__init__() missing 1 required positional argument: 'estimators'`, because structural ensemble config (base estimators, final estimator, voting strategy, CV folds) was leaking into the hyperparameter search space and overriding the already-resolved estimator instance.

Ensemble training jobs (Voting/Stacking Classifier/Regressor) now get their own **Ensemble** category throughout the job-tracking UI instead of being silently folded into Classification/Regression: the Job History drawer, the full Jobs page, and the Experiments page filter dropdown all gain a dedicated Ensemble tab/option (with a Classification/Regression sub-filter, since an ensemble job is still ultimately one or the other underneath). The Job Details view's header now shows a small badge (e.g. "Voting · Classification") identifying the ensemble strategy and underlying task at a glance.

**Two more `skyulf-core` modeling bugs fixed** — both in `StatefulEstimator`'s dataset-shape detection (`modeling/base.py`), the same code path responsible for telling `(X, y)` tuples apart from `(train_df, test_df)` tuples and pulling the target column out of a combined frame:

1. **Target leakage in `_extract_xy_from_tuple`**: when a caller passed `(X, y)` with `y` already non-`None` but `X` was a DataFrame that *also* still contained the target column, the target column was left in `X` instead of being dropped — meaning the target leaked into the feature set for any downstream fit. The check now always strips the target column out of `X` whenever it's present, regardless of whether `y` was already supplied separately.
2. **`(X, y)` tuples misclassified as `(train_df, test_df)`**: the tuple-type detection only checked whether the *first* element was a DataFrame containing the target column, so an `(X, y)` pair where `y` also happened to be a DataFrame (e.g. a single-column target DataFrame rather than a Series) was wrongly treated as a pre-split `(train_df, test_df)` pair — silently training and evaluating on the wrong halves of the data. Detection now also requires the *second* element to be a DataFrame before assuming a pre-split pair, so a DataFrame `y` no longer causes a misclassification.

Both were caught and fixed during a follow-up whole-repo review; regression tests were added to `skyulf-core/tests/test_modeling_base.py` covering both scenarios, and the full `skyulf-core` (2751 tests) and `backend` (1002 tests) suites pass with no regressions.

**Duplicate tuned-model artifact save removed**: fixed-mode ensemble/tuning jobs were calling `_finalize_training_artifacts()` (which persists the trained estimator + bundles transformers for inference) once explicitly and then a second time implicitly later in the same execution path, wasting a redundant disk/S3 write and duplicate artifact registration per job with no functional benefit.

**Backend reliability fixes** (non-ML, surfaced by the same review pass): a race condition in the Celery task DB engine's lazy singleton initialization (two concurrent tasks could each create their own `sync_engine`/`session_factory`) is now guarded with a lock; the data-ingestion status endpoint no longer returns a bare `{"status": "unknown"}` with missing `progress`/`updated_at` fields when a source's stored status is malformed or absent; and the frontend's job-polling aggregate status no longer gets stuck reporting `"running"` forever when every polled job has already reached an unsuccessful terminal state.

**Security hardening**: a security assessment found that the S3 data-source connector and catalog (`data_ingestion/connectors/s3.py`, `data/catalog.py`) accepted a caller-supplied `endpoint_url`/`aws_endpoint_url` in per-request `storage_options` with no validation, letting a request redirect the server's outbound S3 client to an arbitrary host (SSRF, including cloud metadata endpoints) — fixed by discarding any caller-supplied endpoint and only ever trusting a server-operator-configured `AWS_ENDPOINT_URL` setting. Separately, `/health/detailed` was disclosing per-backend/integration status (DB, cache, third-party service names) to unauthenticated callers — it now reports a single aggregate `dependencies_healthy` boolean instead.

## v0.7.0

Kicks off the 0.7.x series with the first model-explainability feature: a SHAP Explainability tab in the Experiments page with four sub-views — a cross-run Summary bar chart (mean |SHAP value| per feature), plus per-run Beeswarm, Dependence, and Waterfall plots for digging into individual predictions, similar to the graph types the `shap` library itself provides. Backend/skyulf-core gain a new optional `shap` dependency and a lightweight, training-time explainability computation (`compute_shap_explanation`) that never blocks or fails a training job if SHAP can't run for a given model/dataset combination.

Adds two more sub-views to round out the SHAP tab: **Force Plot** (a compact, single-row version of Waterfall — same per-row `base_value`/`shap_values` data, no new computation needed) and **Interaction Plot** (a heatmap of the top feature-pairs' joint influence on predictions, via `shap.TreeExplainer.shap_interaction_values` — tree-based models only, e.g. Random Forest/Gradient Boosting/XGBoost; gracefully omitted for linear/kernel models). `compute_shap_explanation` now returns an additional `interactions` key (`{feature_names, matrix} | None`).

Also fixes chart readability in dark mode across the Dashboard, EDA page, and the new SHAP views: chart colors are now centralized in a single reactive theme module (`core/theme/chartTheme.ts` + `useChartTheme()` hook) instead of each component computing colors ad hoc, so charts update live when the theme is toggled rather than only at first render. This also fixed a real bug where pie chart tooltips (Dashboard, EDA overview) rendered black, unreadable text on a dark background — Recharts' default tooltip item/label styles default to black unless explicitly overridden, which is now handled centrally.

Adds **Segmentation** as a first-class, trainable model type (previously clustering only existed as a fixed, non-configurable KMeans(k=3) add-on inside EDA profiling). A new `kmeans` model (skyulf-core `modeling/clustering.py`) plugs into the existing training/evaluation pipeline with zero changes to the sklearn wrapper layer, using an empty `target_column` as the established "no target" sentinel. Evaluation gains clustering-specific metrics (silhouette, Calinski-Harabasz, Davies-Bouldin scores) and a cluster summary (sizes + per-cluster centroids/mean feature values) via a new `evaluate_clustering_model()`. The frontend gets a dedicated **Segmentation** canvas node (reuses the Basic Training node's model/hyperparameter UI, with the target-column selector and Cross-Validation section hidden since neither applies to unsupervised clustering) and a new **Segmentation** tab in the Experiments page showing cluster-quality score cards, a cluster-size bar chart, and per-cluster centroid cards. Advanced Tuning's model dropdown now excludes clustering algorithms, since tuning requires a supervised scoring metric that doesn't apply to unsupervised clustering.

Rounds out Segmentation with three more deployable clustering algorithms — **Mini-Batch K-Means** (faster K-Means variant for large datasets), **Gaussian Mixture** (models each cluster as its own Gaussian shape, handles elongated/overlapping clusters via `covariance_type`), and **Birch** (memory-efficient tree-based summarization, good for very large datasets) — chosen because, like K-Means, all three implement genuine out-of-sample `.predict()` needed for deployment/inference (unlike DBSCAN/Agglomerative/OPTICS, which only support `fit_predict()` on the training data and remain unsupported).

Also adds two ways to make sense of otherwise-arbitrary cluster IDs, both surfaced in the Segmentation tab and in `skyulf-core`'s evaluation output:

- **Automatic profile labels** — every cluster centroid now gets a `profile` string (e.g. `"High income, Low age"`), computed via a z-score comparison of that cluster's centroid against the dataset-wide mean/std for each feature. No configuration needed; it's just part of `centroid.profile`.
- **Optional reference column** — a new `reference_column` node param (e.g. a known label like species name, which can be textual) is excluded from training entirely — by column name, regardless of dtype, so a numeric label column can't leak into clustering distance either — but is used to compute a cluster → label crosstab (`evaluation.clustering.reference_crosstab`), e.g. "Cluster 0 is 92% setosa". The bundled inference artifact's persisted `feature_columns` list also excludes the reference column, so deployment/manual-prediction correctly expects one fewer feature than the raw training data — fixing a feature-count mismatch (`"X has N features, but KMeans is expecting N-1 features"`) that could otherwise occur at predict time.

The Segmentation node's UI gains a "Reference Column" dropdown (optional, dataset-schema-driven) alongside the existing model/hyperparameter picker, and the Experiments page's Segmentation tab shows the new profile labels on each centroid card plus a "Reference Column Breakdown" section when a reference column was set.
