# Changelog â€” 0.7.x Series

## v0.7.1

Internal refactor: extracted the duplicated cancel/log-append/status-update logic shared by `BasicTrainingManager` and `AdvancedTuningManager` into a common `TrainingJobManagerBase` helper. No behavior, API, or DB schema changes â€” first step of unifying Basic Training and Advanced Tuning into a single flow.

Continues the unification with the orchestration and execution layers: `StepType` gains a single canonical `training` value (with `basic_training`/`advanced_tuning` retained as backward-compatible aliases so already-saved pipeline configs and job rows keep executing unchanged), and the pipeline engine now dispatches every training node through one path driven by a `run_mode` param (`"fixed"` for a single configured model, `"tuned"` for a multi-candidate hyperparameter search) instead of two separate step handlers. `run_mode="fixed"` reuses the exact same tuning-engine machinery as `"tuned"`, just degenerated to a single trial, so both flows now share one code path end-to-end from node config to trained model.

Rounds out the frontend to match: a single generic `TrainingNode`/`TrainingSettings` component (keyed by `run_mode: 'basic' | 'advanced'`) replaces the separate Basic Training / Advanced Tuning settings UIs, reusing the same hyperparameter, search-space, and strategy widgets either way. Four task-scoped node wrappers â€” **Classification**, **Regression**, **Text Classification**, and **Segmentation** â€” now appear in the node palette (derived from each model's registry tags), replacing the old generic Basic Training/Advanced Tuning node choice; the legacy nodes are kept registered but hidden so previously-saved canvases keep loading with zero migration. The Experiments page's job-history filters and comparison view were similarly updated to filter by task type instead of basic/advanced, including a fix so all four clustering algorithms (not just K-Means) correctly surface under the Segmentation tab.

Finishes the unification on the database side: `BasicTrainingJob` and `AdvancedTuningJob` â€” previously two parallel tables with duplicated columns â€” are replaced by a single `TrainingJob` table discriminated by `run_mode` (`"fixed"`/`"tuned"`), with `run_number`/`version` unified into one `version` column. Every job-facing code path (job service, `/jobs` and `/stats` endpoints, model registry version counting, monitoring drift/slow-node scans) now reads from the one table scoped by `run_mode` instead of querying two tables and merging results. This is a schema-only change with no data migration â€” the dev database was dropped and recreated rather than backfilling old rows â€” so it only affects fresh environments; `BasicTrainingManager`/`AdvancedTuningManager` remain separate manager/strategy classes for now (planned for later cleanup once this is stable).

## v0.7.0

Kicks off the 0.7.x series with the first model-explainability feature: a SHAP Explainability tab in the Experiments page with four sub-views â€” a cross-run Summary bar chart (mean |SHAP value| per feature), plus per-run Beeswarm, Dependence, and Waterfall plots for digging into individual predictions, similar to the graph types the `shap` library itself provides. Backend/skyulf-core gain a new optional `shap` dependency and a lightweight, training-time explainability computation (`compute_shap_explanation`) that never blocks or fails a training job if SHAP can't run for a given model/dataset combination.

Adds two more sub-views to round out the SHAP tab: **Force Plot** (a compact, single-row version of Waterfall â€” same per-row `base_value`/`shap_values` data, no new computation needed) and **Interaction Plot** (a heatmap of the top feature-pairs' joint influence on predictions, via `shap.TreeExplainer.shap_interaction_values` â€” tree-based models only, e.g. Random Forest/Gradient Boosting/XGBoost; gracefully omitted for linear/kernel models). `compute_shap_explanation` now returns an additional `interactions` key (`{feature_names, matrix} | None`).

Also fixes chart readability in dark mode across the Dashboard, EDA page, and the new SHAP views: chart colors are now centralized in a single reactive theme module (`core/theme/chartTheme.ts` + `useChartTheme()` hook) instead of each component computing colors ad hoc, so charts update live when the theme is toggled rather than only at first render. This also fixed a real bug where pie chart tooltips (Dashboard, EDA overview) rendered black, unreadable text on a dark background â€” Recharts' default tooltip item/label styles default to black unless explicitly overridden, which is now handled centrally.

Adds **Segmentation** as a first-class, trainable model type (previously clustering only existed as a fixed, non-configurable KMeans(k=3) add-on inside EDA profiling). A new `kmeans` model (skyulf-core `modeling/clustering.py`) plugs into the existing training/evaluation pipeline with zero changes to the sklearn wrapper layer, using an empty `target_column` as the established "no target" sentinel. Evaluation gains clustering-specific metrics (silhouette, Calinski-Harabasz, Davies-Bouldin scores) and a cluster summary (sizes + per-cluster centroids/mean feature values) via a new `evaluate_clustering_model()`. The frontend gets a dedicated **Segmentation** canvas node (reuses the Basic Training node's model/hyperparameter UI, with the target-column selector and Cross-Validation section hidden since neither applies to unsupervised clustering) and a new **Segmentation** tab in the Experiments page showing cluster-quality score cards, a cluster-size bar chart, and per-cluster centroid cards. Advanced Tuning's model dropdown now excludes clustering algorithms, since tuning requires a supervised scoring metric that doesn't apply to unsupervised clustering.

Rounds out Segmentation with three more deployable clustering algorithms â€” **Mini-Batch K-Means** (faster K-Means variant for large datasets), **Gaussian Mixture** (models each cluster as its own Gaussian shape, handles elongated/overlapping clusters via `covariance_type`), and **Birch** (memory-efficient tree-based summarization, good for very large datasets) â€” chosen because, like K-Means, all three implement genuine out-of-sample `.predict()` needed for deployment/inference (unlike DBSCAN/Agglomerative/OPTICS, which only support `fit_predict()` on the training data and remain unsupported).

Also adds two ways to make sense of otherwise-arbitrary cluster IDs, both surfaced in the Segmentation tab and in `skyulf-core`'s evaluation output:

- **Automatic profile labels** â€” every cluster centroid now gets a `profile` string (e.g. `"High income, Low age"`), computed via a z-score comparison of that cluster's centroid against the dataset-wide mean/std for each feature. No configuration needed; it's just part of `centroid.profile`.
- **Optional reference column** â€” a new `reference_column` node param (e.g. a known label like species name, which can be textual) is excluded from training entirely â€” by column name, regardless of dtype, so a numeric label column can't leak into clustering distance either â€” but is used to compute a cluster â†’ label crosstab (`evaluation.clustering.reference_crosstab`), e.g. "Cluster 0 is 92% setosa". The bundled inference artifact's persisted `feature_columns` list also excludes the reference column, so deployment/manual-prediction correctly expects one fewer feature than the raw training data â€” fixing a feature-count mismatch (`"X has N features, but KMeans is expecting N-1 features"`) that could otherwise occur at predict time.

The Segmentation node's UI gains a "Reference Column" dropdown (optional, dataset-schema-driven) alongside the existing model/hyperparameter picker, and the Experiments page's Segmentation tab shows the new profile labels on each centroid card plus a "Reference Column Breakdown" section when a reference column was set.
