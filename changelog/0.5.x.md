# Changelog — 0.5.x Series

## Unreleased

### 🐛 Bug Fixes
- **Merge Everywhere:** Multi-input merging now applies to *all* node types (preview, transformers, feature engineering), not just training/tuning. Previously, a node with two parents silently kept only the first input — so connecting two preprocessing branches into a Preview, Deduplicate, etc. dropped one branch entirely.
- **SplitDataset Preserved Through Merge:** When all inputs to a downstream node are `SplitDataset`s, the merge now produces a `SplitDataset` keeping `train`, `test`, and `validation` halves instead of flattening to train-only. Fixes silently missing `test_*` metrics when training merged after a Splitter → parallel preprocessing fan-out.
- **Index-Aware Column Merge:** `_merge_frames` now resets indices before column-wise concatenation, preventing NaN rows when upstream branches kept different row indices.
- **Last-Wins Column Merge (Bug 9i):** `_merge_frames` now uses last-wins semantics on overlapping columns instead of first-wins. A scaler placed *after* a splitter feeding back into a downstream node now actually contributes its scaled columns rather than being silently dropped because the unscaled sibling appeared first in the input list.
- **(X, y) Tuple Preserved Through Splitter (Bug 9c):** `SplitApplier.apply` now detects `(X, y)` tuple inputs and a `target_column` present in plain DataFrame inputs, and emits clean `SplitDataset` slots holding `(X, y)` tuples per split. Downstream transformers no longer see `(df, None)` placeholders that broke training.
- **Duplicate Edge Dedup:** `_resolve_all_inputs` deduplicates repeated input IDs while preserving order, so a canvas that emits two edges between the same two nodes no longer triggers spurious sibling-fan-in or double-merge behavior.
- **Redundant-Edge Suppression in Sibling Fan-In Warning:** `_merge_inputs` no longer warns when one input is a transitive ancestor of another (e.g. `Splitter` feeding both `Encoder` directly **and** via `Scaler`). That's a harmless redundant edge — the descendant supersedes under last-wins, so the merge is effectively just "use the descendant". Warning still fires for true sibling fan-in (independent branches sharing only a deeper common ancestor).

### ✨ Features
- **Sibling Fan-In Advisory (Bug 9j):** `_merge_inputs` detects when two or more inputs to the same node share a transitive ancestor (e.g. `Scaler` and `Encoder` both forked off the same `Splitter`) and records a `sibling_fan_in` advisory. Warnings flow through `PipelineExecutionResult.merge_warnings` → `PreviewResponse.merge_warnings` so the UI can surface a "column union + last-wins" hint instead of users guessing why one branch's columns silently overwrite the other's.
- **Merge Warning Detail Fields:** Engine now reports `overlap_columns` (the actual columns subject to last-wins overwrite) and `winner_input` (which input's values won) on every `sibling_fan_in` advisory, so the UI can name the concrete columns at risk and the branch that "won" instead of generic phrasing.
- **Collapsible Merge Advisory Banner:** `ResultsPanel` collapses merge warnings to a single-line summary by default (`▶ ⚠ N merge advisories — Node A, Node B`). Click to expand for full per-warning detail (inputs, overlap columns, winner, fix hint). Reduces visual noise on canvases with multiple advisories while keeping them one click away.
- **Merge Badge Severity Coloring:** `CustomNodeWrapper` merge badge turns amber with a ring when the node has an active `sibling_fan_in` advisory with `overlap_columns`; stays blue otherwise. Makes risky fan-in visible at a glance on the canvas.
- **Auto-Fix "Chain instead" Button:** Each expanded merge advisory row with `overlap_columns` gets a Wand2-icon button that rewires the fan-in into a linear chain via the new `chainSiblings` store action. Keeps the head sibling's upstream edges, strips the tail siblings' incoming edges, and re-routes through registry handles so the resulting graph is a clean linear pipeline.

### 📚 Docs
- **Troubleshooting → Merge Advisory:** New section in `docs/user_guide/troubleshooting.md` explaining what the *"X merges N parallel branches…"* banner means, why overlapping-column fan-in silently loses data, the canvas steps to chain instead, and a what-works/what-doesn't reference table.

### 🛠 Tooling / CI
- **Type checker:** Replaced mypy with [`ty`](https://github.com/astral-sh/ty) (Astral, Rust-based). New CI step `ty check backend skyulf-core/skyulf skyulf-core/tests …` runs in a fraction of mypy's time. `[tool.ty]` config added to `pyproject.toml`; `mypy.ini` kept only as an editor fallback.
- **Formatter:** Added `black --check` step to CI (line-length 100, configured in `[tool.black]`). Run `black <paths>` locally to auto-fix before pushing.
- **`requirements-ci.txt`:** Swap `mypy` → `ty`, add `black`.

### 🧪 Tests
- **`tests/test_merge_everywhere.py`:** 9 unit tests covering SplitDataset merge, preview merging from both Dataset and Splitter sources, transformer multi-input merge, and training-with-test-metric preservation after a Splitter merge.
- **`tests/test_merge_scenarios_e2e.py`:** 9 end-to-end scenarios (sequential chain, fan-in last-wins, X/y preservation, split→scaler, sibling fan-in warning, multi-branch training, duplicate-edge dedup, full 3-path canvas). Each scenario writes a JSON proof to `tests/artifacts/merge_scenarios/` recording every node's output shape, the engine's merge warnings, and per-node status — so regressions show up as both a failed assertion and an inspectable artifact.
- **Redundant-Edge Test Alignment:** `test_scenario_02_fan_in_to_transformer_lastwins` now asserts that the Splitter→Scaler+Splitter→Downstream pattern does **not** fire `sibling_fan_in` (matches the new redundant-edge filter). `test_scenario_05_sibling_fanin_warning` asserts the new `winner_input` and `overlap_columns` fields are populated correctly.

## v0.5.1 — Branch-Aware Preview Results

### 🎨 Frontend (UX Polish)
- **Readable Applied-Steps:** `ResultsPanel` pills now show node display names (e.g. `Drop Rows`, `Standard Scaler`) instead of raw UUIDs. A `nodeLabelMap` resolves each node ID against the canvas (`data.label || data.title || prettified definitionType`) and falls back to the ID only when no node is found.
- **Auto-Layout ("Tidy") Button:** New toolbar button (Wand2 icon) runs `dagre` left-to-right layout over the current canvas to auto-arrange multi-branch pipelines that get visually chaotic. Lives in `core/utils/autoLayout.ts` and reuses the existing `dagre` dependency.

### 📚 Docs
- **Kahn's + BFS Explainer:** `_collect_ancestors()` docstring rewritten with a step-by-step explanation of BFS (FIFO queue, level-by-level neighbour expansion) and why both BFS and Kahn's are needed (BFS for *which* nodes, Kahn's for *what order*). `docs/architecture.md` got a matching one-line BFS callout.

### 🔧 Backend
- **Multi-Branch Preview:** `POST /api/pipeline/preview` now partitions parallel pipelines via `partition_parallel_pipeline()` and runs each branch through a shared artifact store. Per-branch preview data is returned in a new `branch_previews` field keyed by branch label (e.g. `"Path A · Random Forest"`).
- **Backward Compatible:** Single-branch pipelines still return the same flat `preview_data`. The new `branch_previews` field is `null` for single-branch and only populated when 2+ branches exist.
- **Refactor:** Extracted preview-data extraction into `extract_preview()` and `pick_target_node_id()` closures inside the endpoint — removes duplicated logic and makes the multi-branch loop straightforward.
- **Helper:** Added `_prettify_model_type()` and `_branch_label()` so backend labels mirror the frontend `useBranchColors` convention exactly.

### 🐛 Bug Fixes
- **Topological Order in `_collect_ancestors`:** Replaced reversed-BFS with Kahn's algorithm restricted to the ancestor subgraph. Reversed BFS produced incorrect orderings on diamond-shaped graphs (e.g. `Dataset → Splitter → Scaler` plus a `Dataset → Scaler` shortcut), causing `FileNotFoundError: Artifact not found` at run time because a child node executed before its parent's artifact was written.
- **Preview Skips Training:** Preview now strips `basic_training` / `advanced_tuning` nodes from each sub-pipeline before execution. Models are no longer fitted during preview — runs are faster, no model-side errors, and the preview target becomes the last data-bearing node automatically. Branch labels still derive from the (un-stripped) original sub-pipeline so the model name appears in the tab.
- **Multi-Leaf Preview:** Added `partition_for_preview()` that splits the graph by **data leaves** rather than by training terminals. A canvas with 3 parallel preprocessing chains and no training nodes now renders 3 branch tabs in `ResultsPanel`. Falls back to a single sub-pipeline when only one data leaf exists.
- **Preview Mirrors Run All Experiments:** When training nodes exist, preview now uses `partition_parallel_pipeline()` so the number of preview tabs matches the number of jobs that "Run All Experiments" would create. Fixes the case where adding a 2nd training node didn't show up as an extra preview tab. `partition_for_preview()` is used only when no training nodes exist.
- **Per-Branch Applied Steps:** `PreviewResponse` now includes `branch_node_ids` (per-branch list of node IDs that actually ran). `ResultsPanel` filters the "applied steps" pill bar by the active branch tab, so `Drop Rows` only shows up under Path A — not under every tab. Falls back to the full list for single-branch responses.

### 🎨 Frontend
- **Branch Tabs:** `ResultsPanel` now renders a top-level branch tab bar (with colored dots from `generateBranchColors`) when the response includes `branch_previews`. Selecting a branch reveals its existing split tabs (train/test/X/y).
- **Header Hint:** Header shows `· N branches` count when multi-branch results are available.
- **Type System:** `client.ts` adds `BranchPreviews` type and `PreviewResponse.branch_previews?` field.
- **Color Reuse:** Exported `generateBranchColors` from `useBranchColors.ts` so the panel and edges share the exact same palette.

## v0.5.0 — Promote Winner & Architecture Improvements

### 🔧 Backend
- **Promote Winner API:** Added `POST /api/pipeline/jobs/{job_id}/promote` and `DELETE /api/pipeline/jobs/{job_id}/promote` endpoints for marking a completed job as the promoted winner (or removing promotion).
- **Database Schema:** Added `promoted_at` (DateTime, nullable) column to the `MLJob` abstract base class — applies to both `BasicTrainingJob` and `AdvancedTuningJob` tables.
- **Job Manager:** New `promote_job()` and `unpromote_job()` static methods on `JobManager` that look up jobs across both tables and set/clear the `promoted_at` timestamp.
- **Schema Extension:** `JobInfo` Pydantic model and both training manager `to_dict` / `JobInfo()` constructors now include `promoted_at`.

### 🎨 Frontend
- **Promote Button:** ExperimentsPage sidebar job cards show a Trophy icon button (amber when promoted, gray when not) next to the Deploy button. Clicking toggles promotion via API with optimistic UI update.
- **Winner Badge:** Promoted jobs display a "Winner" badge (amber pill with Trophy icon) in the sidebar card and in the comparison table's Model Type row.
- **Promoted Sort:** Promoted jobs float to the top of the experiments sidebar list.
- **Store Actions:** `useJobStore` gains `promoteJob` and `unpromoteJob` actions with optimistic state updates.
- **API Client:** `jobsApi.promoteJob()` and `jobsApi.unpromoteJob()` methods added.

### 🔀 Multi-Path Pipeline (Phase 1 & 2 Completion)
- **Connection Validation Warnings (1f):** `useGraphStore.onConnect()` now warns when connecting a 2nd+ input to any node — explains auto-merge behavior for training nodes, and warns about single-input-only preprocessing nodes.
- **Concurrent Branch Execution (2b):** Non-Celery path now uses `ThreadPoolExecutor` to run parallel branches concurrently instead of sequentially. Celery path was already concurrent.
- **Branch Metadata (2c):** `JobInfo` schema now includes `branch_index` and `parent_pipeline_id` fields, parsed from the branch pipeline_id suffix (e.g. `pipeline__branch_0`).
- **Branch Comparison Card:** ExperimentsPage shows a "Parallel Branch Comparison" card when selected jobs share a `parent_pipeline_id` — side-by-side metrics table with best-value highlighting (★) and branch color dots.
- **Branch Labels:** Sidebar cards and detail table show a purple "Branch N" badge when the job is part of a parallel run.

### 🔀 Connected Components — Experiment Group Isolation
- **Disconnected Subgraph Detection:** Added `_split_connected_components()` in `graph_utils.py` — uses BFS to detect disconnected subgraphs on the canvas. Each subgraph gets its own unique `pipeline_id` so only branches sharing data lineage are grouped in the same experiment.
- **Caller Integration:** `api.py` `run_pipeline` now calls `_split_connected_components()` before `partition_parallel_pipeline()`, ensuring isolated datasets produce separate experiment groups.
- **Tests:** 5 new tests covering fully connected, disconnected, mixed, end-to-end, and empty graph scenarios (18 total).

### 📊 Best Score Metric Label
- **Backend:** `scoring_metric` field now included in tuning result (`engine.py`, `advanced_tuning_manager.py`) so the frontend knows which metric the best score refers to.
- **Core:** `TuningResult` dataclass now carries `scoring_metric` set to the actual resolved sklearn scorer name (e.g., `f1_weighted` for multiclass, not the original user selection `f1`).
- **Frontend:** All "Best Score" / "Score" labels across JobsDrawer, ExperimentsPage, BasicTrainingSettings, and AdvancedTuningSettings now display the actual metric name (e.g., "Best Score (F1 Weighted)" instead of generic "Best Score").
- **Shared Utility:** Extracted `formatMetricName()` into `core/utils/format.ts` — single source of truth for metric display names, with full coverage of sklearn scorer strings (e.g., `neg_mean_squared_error` → "MSE").

### 🎨 UX: Jobs Drawer Tab Order
- **Tab Swap:** Standard Training tab now appears first (left), Advanced Training second (right) in the JobsDrawer panel.
