### 🎨 Frontend — Interactive Probability Threshold in Evaluation View
- **Threshold slider** added to the evaluation view controls bar (0.01–0.99, default 0.50). Applies one-vs-rest (OvR) prediction logic: if `P(selectedClass) ≥ threshold`, predicts that class; otherwise takes the argmax of the remaining classes. Works for both binary and multiclass.
- **Confusion matrix** now re-derived in real-time from raw `y_proba` at the selected threshold (instead of stored `y_pred`). The label shows the active threshold (`@ t=0.45`). All splits (train/test/validation) update simultaneously.
- **Live metric tiles** (Accuracy, Precision, Recall, F1) displayed below each confusion matrix, computed via OvR for the selected class. Update instantly as the slider moves.
- **ROC curve operating point**: a red dot marks the current (FPR, TPR) position at the selected threshold, so users can see the trade-off directly on the curve.
- **Controls bar made sticky** (`sticky top-0 z-10`): the split toggles, class selector, threshold slider, and mapping annotation stay pinned at the top while scrolling through train/test/validation split charts.
- **Controls layout tightened**: all controls now on a single horizontal bar (split toggles + divider + class dropdown + threshold slider + mapping annotation), replacing the previous two-row stacked layout. Frees vertical space and removes cramping on smaller screens.

### 🤖 Core — New Evaluation Metrics
- **`balanced_accuracy`** added to classification: like accuracy but averages per-class accuracy equally — not skewed by majority class size.
- **`matthews_corrcoef` (MCC)** added to classification: single-number summary of the confusion matrix; best metric for imbalanced binary classification (+1 = perfect, 0 = random, −1 = inverse).
- **`log_loss`** added to classification: computed whenever `predict_proba` is available; penalises confident wrong predictions.
- **`explained_variance`** added to regression: fraction of target variance explained, similar to R² but does not penalise systematic bias.

### 🤖 Core — Tuning Engine

### 🐛 Bug Fixes — CMA-ES / Optuna
- **Missing `cmaes` package:** `optuna`'s `CmaEsSampler` requires the optional `cmaes` package which is not installed automatically with `optuna`. Added `cmaes>=0.10` to `requirements-fastapi.txt`; pipeline runs no longer crash with `ModuleNotFoundError: No module named 'cmaes'`.
- **CMA-ES distribution mapping:** Numeric search-space lists (e.g. `[100, 200, 500]`) are now converted to `IntDistribution` / `FloatDistribution` instead of `CategoricalDistribution`. CMA-ES only handles continuous distributions; categorical parameters caused a silent fallback to `RandomSampler` for every param. String/bool/None lists remain `CategoricalDistribution`.
- **`warn_independent_sampling=False`:** Added to `CmaEsSampler` construction to suppress the unavoidable Optuna warning for genuinely categorical params that cannot be mapped to a continuous distribution.
- **Frontend CMA-ES warning:** `StrategySettingsModal` now shows an amber warning when the user selects CMA-ES for a model with string/boolean parameters (logistic regression, random forest, SVC, SVR, k-NN, decision tree, ridge, linear), explaining that those params will be sampled randomly and suggesting TPE or XGBoost/GradientBoosting instead.
- **`uv` overwrote `skyulf-core` with PyPI stale release:** `uv add` was resolving `skyulf-core>=0.1.2` to the PyPI `0.1.2` release and replacing the local editable `0.1.14`. Fixed by adding `[tool.uv.sources]` to root `pyproject.toml` pinning `skyulf-core` to `path = "./skyulf-core", editable = true`.
- **`requires-python` bumped to `>=3.11`:** Root `pyproject.toml` declared `>=3.10`, forcing `uv` to lock `scikit-learn 1.7.2` (the last version supporting Python 3.10). Models saved in the `skyulf-core` dev env (Python 3.12) were pickled with `sklearn 1.8.0`, causing `InconsistentVersionWarning` on every artifact load. Bumping to `>=3.11` lets `uv` resolve `scikit-learn 1.8.0`, eliminating the mismatch.
- **`imbalanced-learn` 0.14.0 → 0.14.1:** sklearn 1.8.0 renamed the private `_is_pandas_df` to `is_pandas_df` in `sklearn.utils.validation`. `imbalanced-learn 0.14.0` imported the old private name, causing a hard `ImportError` on startup. Updated to `0.14.1` (the patch release that fixes this). Bumped floor pins in `requirements-fastapi.txt` and `requirements-ci.txt` accordingly.
- **Python version floor raised to 3.12 everywhere:** Further narrowed from 3.11 to 3.12 — the only version in active use. Updated `requires-python`, classifiers, README badge, CONTRIBUTING, QUICKSTART, docs, CI (`codeql.yml`), and `index.html`. Removed packages that were only needed for Python <3.12 (`async-timeout`, `importlib-metadata`, `tomli`, `zipp`) from the lock.

### 🐛 Bug Fixes — Frontend polling spam
- **Constant `/api/pipeline/jobs` polling:** `toggleDrawer(true)` was calling `startPolling()`, so opening the experiments panel or jobs drawer started a 3-second poll loop that ran forever. Fixed: opening the drawer now only calls `fetchJobs()` once; sustained polling only starts on job submission.
- **Orphaned jobs blocked polling stop:** Jobs stuck in `running`/`queued` from a killed previous session caused `hasActive` to remain `true` indefinitely, preventing `stopPolling()` from ever being reached. Fixed by treating a job as active only if it was created within the last 2 hours.
- **Hard polling deadline:** Even if a fresh job somehow stays stuck, a 30-minute maximum cap (`MAX_POLL_DURATION_MS`) ensures the poll interval is cleared unconditionally.
- **`useNodeJobSummaries` safety-net gated on active jobs:** The `/api/pipeline/jobs/node-summaries` endpoint was also being polled every 30 s unconditionally while the canvas was mounted. The safety-net `setInterval` now only starts when there are fresh active jobs and is cleared as soon as none remain. Idle canvas = zero background requests.
### 🐛 Bug Fixes — LightGBM warnings
- **`[LightGBM] [Warning] No further splits with positive gain`:** LightGBM 4.x emits this from native C++ stderr; the Python-level `verbose=-1` / `verbosity=-1` constructor params do **not** silence it. Fixed by registering a no-op logger via `lightgbm.register_logger()` at module load in both `skyulf/modeling/classification.py` and `regression.py`. Constructor `verbosity=-1` retained as a defense-in-depth.
- **`UserWarning: X does not have valid feature names, but LGBMClassifier was fitted with feature names`:** LightGBM 4.x exposes `feature_names_in_` as a property whose deleter is intentionally a no-op (upstream `LGBMModel.feature_names_in_.deleter` literally just `pass`es). Deleting the attribute is therefore impossible. Fixed by overriding `predict()` / `predict_proba()` on `LGBMClassifierApplier` and `predict()` on `LGBMRegressorApplier` to wrap the call in `warnings.catch_warnings()` filtering this exact message. The same filter remains in `TuningCalculator.tune()` to catch the warning during sklearn's internal CV `score()` calls (HalvingGrid/HalvingRandom/Optuna).
- **Cross-model audit:** Verified that all 12 classifiers and 14 regressors registered in `NodeRegistry` (LogisticRegression, RandomForest, SVC, KNN, DecisionTree, GradientBoosting, AdaBoost, XGBoost, ExtraTrees, HistGradientBoosting, LGBM, GaussianNB and their regressor counterparts + Linear/Ridge/Lasso/ElasticNet/SVR) emit zero warnings and zero stderr through the calculator/applier path — only LightGBM had this quirk.

### 🐛 Bug Fixes
- **Stop Job didn't actually stop training (Celery path).** Cancelling a running training/tuning job from the UI flipped the DB row to `cancelled` but the Celery worker kept training and, on completion, overwrote the row back to `completed` via `update_status_sync` — making Stop look like a no-op. Three coordinated fixes: (1) [`/pipeline/run`](backend/ml_pipeline/api.py) now captures the `AsyncResult` from `run_pipeline_task.delay(...)` and persists `task.id` onto the job's `job_metadata.celery_task_id` via a new [`JobManager.attach_celery_task_id`](backend/ml_pipeline/execution/jobs.py) helper. (2) [`BasicTrainingManager.cancel_training_job`](backend/ml_pipeline/execution/basic_training_manager.py) and [`AdvancedTuningManager.cancel_tuning_job`](backend/ml_pipeline/execution/advanced_tuning_manager.py) now call `celery_app.control.revoke(task_id, terminate=True, signal="SIGTERM")` after marking the row cancelled, which actually kills the in-flight `model.fit`. (3) Both managers' `update_status_sync` now refuses to flip a row away from `cancelled` (logs are still appended so cancellation traces survive) — protects against late writes that race past the revoke, and also means the non-Celery / thread-pool path no longer regresses Stop into Completed even though Python threads can't be hard-killed. No schema migration required (uses the existing JSON `job_metadata` column).
- **Stop Job still completed on the threads (non-Celery) path.** Follow-up to the fix above: even with the `update_status_sync` guard in place, the worker in [`tasks.py`](backend/ml_pipeline/tasks.py) writes its final `job.status = "completed"` and `strategy.handle_success(job, result)` directly on the SQLAlchemy row — bypassing `update_status_sync` entirely. So when `USE_CELERY=false` (background threads can't be hard-killed), a cancelled tuning job would still finish, write metrics + best params, and flip the row back to completed. Fixed by `session.refresh(job, ["status"])` immediately before the success/failure branch and short-circuiting if the row is already `cancelled` (mirrored in the exception handler so a crash during cancelled cleanup doesn't regress to `failed`). Result: cancellation now sticks across both Celery and threads execution modes.
- **Run Preview dropped parallel branches with shorter chains.** The toolbar's Run Preview only showed data from one branch (typically the "first created") whenever the canvas had multiple parallel preprocessing chains and no training/data-preview terminal — branches with shorter chain lengths were silently pruned. Root cause: the depth-inferred-seed fallback in [`pipelineConverter.ts`](frontend/ml-canvas/src/core/utils/pipelineConverter.ts) filtered leaves to the deepest only, so anything ending with fewer steps got removed before the request hit the backend (where `partition_for_preview` would have correctly produced one tab per branch). Fix: when no explicit terminals are present, keep **all** leaves as seeds. Regression test added to `pipelineConverter.test.ts`.
- **Run Preview hid dangling preprocessing branches when a training node was also on the canvas.** Mixed canvases (e.g. one branch wired to Advanced Training plus 1–N standalone Encoding/Scaling chains) only rendered the training-fed tab in Preview Results. Fixed across three layers: (1) the frontend converter now seeds the reverse-walk with **both** explicit terminals and any data leaf, so dangling branches survive into the request; (2) the `/preview` endpoint in [`backend/ml_pipeline/api.py`](backend/ml_pipeline/api.py) now appends preview-only sub-pipelines (from `partition_for_preview`) to the training-derived ones, deduplicated by leaf id; (3) the canvas-side branch coloring in [`useBranchColors.ts`](frontend/ml-canvas/src/core/hooks/useBranchColors.ts) treats data leaves as pseudo-terminals so they get their own color + edge label.
- **Branch tab labels were unreadable for preview-only paths.** Backend `_branch_label` previously suffixed with the raw upstream node id (e.g. `Path B · scaler-7f3a`). Now uses, in priority order: model name → frontend-supplied `_display_name` (the canvas-shown label like "Encoding" / "Scaling") → friendly version of the leaf step type (`StandardScaler` → `Standard Scaler`). The frontend converter forwards each node's registry/canvas label as `_display_name` so backend tabs and canvas edges read identically.
- **All canvas edge labels rendered as "Path A · …" when multiple terminals existed.** `useBranchColors` used a per-terminal `localIndex` for the path letter, which is always 0 for single-input branches — so a canvas with several terminals (training + dangling preview leaves) showed `Path A · Xgboost`, `Path A · Encoding`, `Path A · Scaling`, … with no way to tell branches apart. Fix: when more than one terminal is present, switch to a global branch index so canvas labels (`Path A`, `B`, `C`, `D`, …) line up with the Preview Results tabs.
- **Preview Results tabs now show `#1`/`#2` disambiguators for sibling training branches.** When two training nodes share the same model_type (e.g. two XGBoost terminals), the canvas already labeled them `Path A · Xgboost #1` / `Path B · Xgboost #2`, but the preview tabs read just `Path A · Xgboost` / `Path B · Xgboost`, breaking the visual mapping. ### 🐛 Bug Fixes (cont.)
- **`DELETE /data/api/sources/{id}` 500'd whenever the source had any EDA reports.** SQLAlchemy's default relationship cascade tried to NULL out `eda_reports.data_source_id` before deleting the parent row, which violated the NOT NULL constraint and rolled the whole transaction back with `IntegrityError: NOT NULL constraint failed: eda_reports.data_source_id`. - **`POST /api/ingestion/{id}/cancel` returned 400 on race conditions.** When the UI's 5 s polling refreshed a row to `completed` between the user opening the menu and clicking Cancel, the click landed on a job whose `ingestion_status` was no longer `pending`/`processing` and the backend returned `400 Bad Request: "Job could not be cancelled"` — surfacing as a scary toast for what is essentially a benign race.
